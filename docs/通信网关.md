## 超级万能AI Agent 产品设计文档 - 补充章节：社交通信网关（优化版）

本文档是对《超级万能AI Agent 产品设计文档（v2.3）》的补充升级，重点解决“多平台社交通信接入是否足够方便、灵活、可持续扩展”的问题。

---

### 1. 结论（审阅结果）

当前版本已经具备以下基础：
- Provider 可插拔架构
- 统一路由规则
- 审批回调链路
- 凭证加密存储

仍需优化的关键点：
1. **平台抽象不完整**：缺少“传输层抽象”（Webhook / WebSocket / Long Polling / Native Bridge）。
2. **消息模型偏弱**：缺少 `event_id/thread_id/reply_to/attachments/stream` 等通用字段。
3. **可靠性机制不足**：未明确去重、幂等、重试、限流、顺序保证。
4. **多平台能力矩阵不足**：仅覆盖少数平台，难以指导批量接入。
5. **接入规范缺少“准入标准”**：新平台接入时缺乏验收清单与自动化回归规范。

---

### 2. 目标

在不改动核心 Agent 引擎的前提下，实现：
- 同一条社交消息可以标准化进入同一个 Agent 会话模型。
- 任意平台可按统一接口接入，支持热插拔与灰度发布。
- 新平台接入从“改核心代码”变成“实现 Adapter + 配置”。

### 4. 统一事件模型（必须）

```go
type InboundEvent struct {
    EventID      string            // 平台事件唯一ID，用于去重
    Platform     string            // telegram / slack / ...
    Protocol     string            // webhook/ws/longpoll/bridge
    EventType    string            // message, callback, join, leave, typing
    SenderID     string
    SenderName   string
    ChatID       string
    ThreadID     string
    ReplyToID    string
    Mentions     []string
    Text         string
    Attachments  []Attachment
    Metadata     map[string]string
    OccurredAt   time.Time
    RawPayload   []byte
}

type OutboundEvent struct {
    MessageID      string
    Platform       string
    ChatID         string
    ThreadID       string
    ReplyToID      string
    TextMarkdown   string
    Actions        []Action
    Attachments    []Attachment
    Stream         bool              // 是否流式输出
    Phase          string            // thinking/typing/executing/final
    IdempotencyKey string            // 出站幂等键
    Priority       string            // low/normal/high/critical
    TTLSeconds     int
}

type Attachment struct {
    Type     string // image,file,audio,video
    URL      string
    FileName string
    MIME     string
    Size     int64
}
```

说明：
- `EventID` + `Platform` 用于入站去重。
- `IdempotencyKey` 用于出站幂等，防止重发风暴。
- `ThreadID` 兼容 Discord/Slack/Matrix 等线程模型。
- `Phase` 用于表达实时状态：`thinking`（正在思考）、`typing`（正在输入）、`executing`（正在执行工具）、`final`（最终回复）。

实时状态事件规范：
```go
type PresenceEvent struct {
    Platform string // telegram/slack/...
    ChatID   string
    ThreadID string
    State    string // thinking | typing | executing | idle
    Message  string // 可选文案，如“正在思考中...”
    TTLMs    int    // 状态显示时长
}
```

网关行为约定：
1. Agent 进入推理阶段时发送 `thinking`。
2. Agent 开始流式输出前发送 `typing`。
3. 工具调用阶段发送 `executing`。
4. 完成回复后发送 `idle` 或 `final`。

### 5. Provider 接口（升级）

```go
type Provider interface {
    Name() string
    Protocol() string                     // webhook/ws/longpoll/bridge
    Capabilities() CapabilityProfile      // 平台能力声明

    Start(ctx context.Context) error
    Stop(ctx context.Context) error

    SetInboundHandler(handler func(context.Context, InboundEvent) error)
    Send(ctx context.Context, event OutboundEvent) (SendResult, error)
    SendPresence(ctx context.Context, event PresenceEvent) error  // 发送实时状态

    Health(ctx context.Context) ProviderHealth
}

type CapabilityProfile struct {
    SupportsEdit          bool
    SupportsButtons       bool
    SupportsThreads       bool
    SupportsTyping        bool
    SupportsAudio         bool
    SupportsStreamingText bool
    MaxTextLen            int
    RateLimitPerMinute    int
}

// SendResult 发送结果
type SendResult struct {
    MessageID    string            // 平台返回的消息 ID
    Platform     string            // 平台名称
    ChatID       string            // 目标会话 ID
    Delivered    bool              // 是否成功送达
    ErrorMessage string            // 错误信息（若失败）
    Metadata     map[string]string // 平台返回的额外元数据
}

// ProviderHealth 健康状态
type ProviderHealth struct {
    Name         string        // Provider 名称
    Status       string        // healthy / degraded / unhealthy
    LastCheckAt  time.Time     // 上次检查时间
    LatencyMs    int64         // 响应延迟（毫秒）
    ErrorRate    float64       // 错误率（0.0 - 1.0）
    ErrorMessage string        // 错误信息（若有）
    Metrics      HealthMetrics // 详细指标
}

type HealthMetrics struct {
    TotalRequests   int64   // 总请求数
    SuccessCount    int64   // 成功数
    FailedCount     int64   // 失败数
    AvgLatencyMs    int64   // 平均延迟
    RateLimitRemain int     // 剩余配额
    RateLimitReset  int64   // 配额重置时间戳
}
```

这能让网关在运行时自动做降级策略：
- 平台不支持按钮时，把 `Actions` 转命令提示。
- 平台长度限制较小（如 Discord 2000），自动分片发送。
- 平台不支持 `typing` 状态时，静默忽略 `SendPresence` 调用。

### 8. 可靠性与可观测（新增必须项）

1. **入站去重**：`(platform,event_id)` 做 24h 去重缓存。  
2. **出站幂等**：`IdempotencyKey` + 平台响应 ID 做幂等写入。  
3. **重试策略**：指数退避 `1s, 2s, 5s, 10s`，最大 5 次。  
4. **限流桶**：按平台/租户/会话三级限流。  
5. **死信队列**：最终失败进入 DLQ，支持后台重放。  
6. **顺序保证**：同一 `(platform,chat_id,thread_id)` 串行发送。  
7. **审计字段**：必须记录 `provider, protocol, event_id, latency_ms, retry_count, final_status`。

### 8.1 流式输出策略

当 `OutboundEvent.Stream = true` 时，网关采用以下策略处理流式输出：

```go
// gateway/stream.go
package gateway

type StreamHandler struct {
    provider Provider
    buffer   *strings.Builder
    lastSent string
    msgID    string
}

func (h *StreamHandler) OnChunk(ctx context.Context, chunk string) error {
    h.buffer.WriteString(chunk)
    current := h.buffer.String()

    // 策略1：编辑模式（平台支持编辑）
    if h.provider.Capabilities().SupportsEdit && h.msgID != "" {
        // 每累积 50 字符或遇到段落结束符时更新
        if len(current)-len(h.lastSent) >= 50 || strings.HasSuffix(chunk, "\n\n") {
            _, err := h.provider.Send(ctx, OutboundEvent{
                MessageID:    h.msgID,
                ChatID:       h.chatID,
                TextMarkdown: current,
            })
            h.lastSent = current
            return err
        }
        return nil
    }

    // 策略2：分片模式（平台不支持编辑）
    // 每 500 字符发送一条新消息
    if len(current)-len(h.lastSent) >= 500 {
        newMsg := current[len(h.lastSent):]
        result, err := h.provider.Send(ctx, OutboundEvent{
            ChatID:       h.chatID,
            TextMarkdown: newMsg,
        })
        if err == nil {
            h.msgID = result.MessageID
            h.lastSent = current
        }
        return err
    }

    return nil
}

func (h *StreamHandler) OnComplete(ctx context.Context) error {
    // 流式结束，发送最终版本
    current := h.buffer.String()
    if current != h.lastSent {
        _, err := h.provider.Send(ctx, OutboundEvent{
            MessageID:    h.msgID,
            ChatID:       h.chatID,
            TextMarkdown: current,
        })
        return err
    }
    return nil
}
```

**平台能力矩阵**：

| 平台 | 支持编辑 | 支持流式 | 推荐策略 |
|------|---------|---------|---------|
| Telegram | ✅ | ❌ | 编辑模式 |
| Discord | ✅ | ❌ | 编辑模式 |
| Slack | ✅ | ❌ | 编辑模式 |
| Feishu | ✅ | ✅ | 原生流式 |
| DingTalk | ✅ | ❌ | 编辑模式 |
| Matrix | ✅ | ❌ | 编辑模式 |

### 8.3 消息编辑策略

```go
// gateway/edit_policy.go
package gateway

type EditPolicy struct {
    MaxEdits       int           // 最大编辑次数（默认 10）
    EditWindow     time.Duration // 编辑窗口（默认 24h）
    EditThreshold  int           // 内容变化阈值（字符数）
}

func (p *EditPolicy) ShouldEdit(original, new string, editCount int, firstSent time.Time) bool {
    // 超过最大编辑次数
    if editCount >= p.MaxEdits {
        return false
    }
    // 超过编辑窗口
    if time.Since(firstSent) > p.EditWindow {
        return false
    }
    // 内容变化太小（如仅 1-2 字符）
    if abs(len(new)-len(original)) < p.EditThreshold {
        return false
    }
    return true
}
```

### 9. 安全与合规

- 凭证仅密文存储，明文仅在内存短暂使用。
- Webhook 必须做签名校验与时间窗校验（重放保护）。
- 桥接类 Provider（WhatsApp/Weixin/IRC/MaixCam）建议运行在隔离进程或独立容器。
- 社交通道权限最小化：仅开必要 scope。
- 敏感内容（如审批命令）默认双因子校验：账号绑定 + 回调签名。

### 10. 新平台接入标准流程（推荐）

1. 定义 `CapabilityProfile`。  
2. 选定 `Protocol`（webhook/ws/longpoll/bridge）。  
3. 完成 `InboundEvent` 映射。  
4. 完成 `OutboundEvent` 发送与降级。  
5. 接入去重与幂等。  
6. 接入健康检查与速率指标。  
7. 通过回归用例：文本、按钮、附件、重试、并发、断线恢复。  
8. 灰度发布：先单租户，再全量。

- `last`

### 13. 工程实现补充（2026-04-24）

为确保产品文档与代码一致，当前版本已落地以下接口与行为：

1. **网关管理 API**
   - `GET/PUT /api/gateway/config`
   - `GET/PUT /api/gateway/providers`
   - `GET/POST /api/gateway/bindings`
   - `DELETE /api/gateway/bindings/{id}`
   - `GET/POST /api/gateway/routes`
   - `DELETE /api/gateway/routes/{name}`
   - `POST /api/gateway/send`
   - `POST /api/gateway/webhook/{provider}`
   - `GET /api/gateway/dlq`
   - `POST /api/gateway/dlq/{id}`（重放）
   - `DELETE /api/gateway/dlq/{id}`

2. **可靠性机制**
   - 入站去重：`(platform,event_id)` 24 小时缓存。
   - 出站幂等：`idempotency_key` 24 小时缓存（成功后写入）。
   - 重试退避：`1s, 2s, 5s, 10s`，最多 5 次。
   - 限流：平台 / 租户(agent) / 会话(platform+chat+thread) 三级限流。
   - DLQ：失败消息进入死信队列，支持后台重放。
   - 顺序：同 `(platform,chat_id,thread_id)` 使用同泳道串行发送。

3. **Webhook 安全校验**
   - Token：`X-Gateway-Token` 或 `Authorization: Bearer ...`。
   - 签名（可选，配置 `signing_secret` 后生效）：
     - `X-Gateway-Timestamp`：Unix 秒时间戳（允许 5 分钟窗口）
     - `X-Gateway-Signature`：`hex(hmac_sha256(signing_secret, timestamp + "." + raw_body))`

4. **通道能力**
   - 内置 `console`、`webhook` Provider。
   - 对 Telegram/Discord/Slack/Feishu/DingTalk/OneBot/Matrix/LINE/WeCom/Teams 等通道，已提供统一 `proxy provider` 适配层，可按配置启用并通过统一模型收发。

### 14. 数据库 Schema

```sql
-- 通道绑定表：记录用户/群组与 Agent 的绑定关系
CREATE TABLE channel_bindings (
    id TEXT PRIMARY KEY,
    agent_id TEXT NOT NULL,
    platform TEXT NOT NULL,
    chat_id TEXT NOT NULL,
    thread_id TEXT,
    binding_type TEXT DEFAULT 'user',  -- user / group / channel
    enabled BOOLEAN DEFAULT true,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    last_active_at DATETIME,
    metadata JSON,
    UNIQUE(platform, chat_id, thread_id)
);

-- 通道配置表：存储平台凭证和配置
CREATE TABLE channel_configs (
    id TEXT PRIMARY KEY,
    platform TEXT NOT NULL UNIQUE,
    provider_type TEXT NOT NULL,        -- webhook / websocket / longpoll / bridge
    credentials_encrypted BLOB,         -- 加密存储的凭证
    config JSON,                        -- 非敏感配置
    enabled BOOLEAN DEFAULT true,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);

-- 路由规则表
CREATE TABLE routing_rules (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    priority INTEGER DEFAULT 100,
    match_conditions JSON NOT NULL,     -- 匹配条件
    action JSON NOT NULL,               -- 执行动作
    enabled BOOLEAN DEFAULT true,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);

-- 消息审计表
CREATE TABLE message_audit (
    id TEXT PRIMARY KEY,
    direction TEXT NOT NULL,            -- inbound / outbound
    platform TEXT NOT NULL,
    provider TEXT NOT NULL,
    protocol TEXT NOT NULL,
    event_id TEXT,
    chat_id TEXT NOT NULL,
    thread_id TEXT,
    idempotency_key TEXT,
    content_hash TEXT,
    latency_ms INTEGER,
    retry_count INTEGER DEFAULT 0,
    final_status TEXT NOT NULL,         -- success / failed / deduped
    error_message TEXT,
    created_at DATETIME NOT NULL,
    INDEX idx_platform_chat (platform, chat_id),
    INDEX idx_created_at (created_at),
    INDEX idx_idempotency (idempotency_key)
);

-- 入站去重缓存表（24h 过期）
CREATE TABLE inbound_dedup (
    platform TEXT NOT NULL,
    event_id TEXT NOT NULL,
    created_at DATETIME NOT NULL,
    PRIMARY KEY (platform, event_id)
);

-- 出站幂等表（24h 过期）
CREATE TABLE outbound_idempotency (
    idempotency_key TEXT PRIMARY KEY,
    platform TEXT NOT NULL,
    message_id TEXT,
    created_at DATETIME NOT NULL
);

-- 死信队列表
CREATE TABLE dead_letters (
    id TEXT PRIMARY KEY,
    platform TEXT NOT NULL,
    chat_id TEXT NOT NULL,
    event JSON NOT NULL,
    error TEXT,
    retry_count INTEGER DEFAULT 0,
    max_retry INTEGER DEFAULT 5,
    status TEXT DEFAULT 'pending',
    created_at DATETIME NOT NULL,
    last_retry_at DATETIME,
    resolved_at DATETIME,
    resolved_by TEXT,
    metadata JSON
);
```

### 15. 配置示例

```yaml
# config/gateway.yaml
gateway:
  enabled: true
  
  dedup:
    ttl: 24h                    # 去重缓存有效期
    
  retry:
    max_attempts: 5
    backoff: [1s, 2s, 5s, 10s, 30s]
    
  rate_limit:
    enabled: true
    global: 1000                # 全局每分钟请求数
    per_platform:               # 按平台限制
      telegram: 30
      discord: 50
      feishu: 100
      
  dlq:
    enabled: true
    max_retry: 5
    retention: 720h             # 30 天
    
  edit_policy:
    max_edits: 10
    edit_window: 24h
    edit_threshold: 10
    
  stream:
    chunk_size: 50              # 流式输出分块大小
    flush_interval: 500ms       # 刷新间隔
    
  providers:
    telegram:
      enabled: true
      protocol: webhook
      token: ${TELEGRAM_BOT_TOKEN}
      
    discord:
      enabled: true
      protocol: websocket
      token: ${DISCORD_BOT_TOKEN}
      intents: [messages, message_content]
      
    feishu:
      enabled: true
      protocol: websocket
      app_id: ${FEISHU_APP_ID}
      app_secret: ${FEISHU_APP_SECRET}
```
