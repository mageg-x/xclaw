# XClaw 文档实现对照审计

更新日期：2026-04-26

本文档用于对照 `docs/PRD.md`、`docs/终端原生体验.md`、`docs/心跳机制.md`、`docs/通信网关.md` 与当前代码实现，避免 PRD 表述超前于代码。

## 审计结论

### 已实现

1. 统一根目录规划已落地。
   - 默认根目录已经是：
     - Windows: `%APPDATA%/xclaw`
     - macOS: `~/Library/Application Support/xclaw`
     - Linux: `~/.xclaw`
   - 并已统一拆分 `config/`、`data/`、`skills/`、`cache/`、`logs/`、`run/`、`tmp/`、`data/workspaces/`。
   - 代码位置：`cli/config/config.go`

2. Markdown 工作区与记忆体系已实现。
   - 已自动创建 `AGENTS.md`、`SOUL.md`、`IDENTITY.md`、`USER.md`、`TOOLS.md`、`MEMORY.md` 与按日期分割的 `memory/YYYY-MM-DD.md`。
   - 代码位置：`cli/workspace/manager.go`

3. 终端原生体验核心能力已实现。
   - CLI 子命令已包含 `start`、`stop`、`restart`、`status`、`doctor`、`logs`、`gateway`、`workspace`、`update`、`tui`、`mcp-server`、`mcp`。
   - TUI 基于 `bubbletea` / `lipgloss` / `bubbles`。
   - 代码位置：`cli/command_cli.go`、`cli/tui.go`

4. 通信网关大部分设计已实现。
   - 已有统一事件模型、附件、回复链路、幂等、去重、DLQ、能力画像、内容适配、流式分片、Provider 健康状态与限流。
   - 代码位置：`cli/gateway/types.go`、`cli/gateway/gateway.go`、`cli/gateway/reliability.go`、`cli/gateway/content_adapter.go`、`cli/gateway/stream.go`、`cli/gateway/health.go`、`cli/gateway/rate_limiter.go`

5. 心跳机制已实现。
   - 已有心跳配置、调度运行、空闲问候、审计记录与网关通知。
   - 代码位置：`cli/heartbeat/runner.go`

6. MCP 客户端实现已存在，不只是 MCP Server。
   - 已支持外部 MCP Server 的注册、初始化、`tools/list`、`tools/call`、超时、缓存、测试、技能侧自动挂载。
   - 代码位置：`cli/mcpclient/manager.go`、`cli/api/mcp_client_api.go`、`cli/mcpregistry/registry.go`、`cli/tools/executor.go`

7. 标准 Skill 包兼容层已经基本落地，并在本次补齐了版本约束。
   - 已支持 `skill.json` / `skill.yaml` / `skill.yml`、依赖安装、来源记录、锁文件 `.xclaw-lock.json`、校验和、技能市场目录合并。
   - 本次新增：依赖版本约束解析与远端版本选择，支持精确版本与范围约束，例如：
     - `dep-a@1.2.3`
     - `dep-a@>=1.2.0,<2.0.0`
     - `dep-a@^1.2.3`
     - `dep-a@~1.2.3`
   - 代码位置：`cli/skills/market.go`
   - 测试位置：`cli/skills/market_test.go`

8. Web 聊天页已经高强度复用 TDesign Chat 能力。
   - 当前已显式使用 `Chatbot`、`ChatList`、`ChatSender`、`ChatMessage`、`ChatActionbar`、`ChatContent`、`ChatLoading`、`ChatMarkdown`、`ChatThinking`、`Attachments`。
   - `ChatEngine` 能力通过 `Chatbot` 的 `chat-service-config`、`registerMergeStrategy`、`message-change`、`sendUserMessage/abortChat/scrollList` 控制面被直接驱动。
   - 引用回复、投票、附件元数据仍由 XClaw 维护业务状态，但引用展示和草稿引用预览已经改为 `ChatList` 语义渲染，而不是自绘纯 DOM。
   - 代码位置：`web/src/views/ChatView.vue`

9. 多模态能力已补齐到当前 PRD 审计缺口范围。
   - 输入：图片视觉分析、文本类文档提取、PDF、Word（`.docx`）、Excel（`.xlsx`）、PowerPoint（`.pptx`）、代码/文档压缩包（`.zip` / `.tar.gz` / `.tgz`）。
   - 输出：`Mermaid`、`ECharts`、`HTML`、云端图片生成。
   - 音频：STT / TTS。
   - 代码位置：`cli/api/capability_api.go`

10. 系统控制入口已补齐到“可执行”级别。
   - 已支持后端 API 触发受控重启。
   - 已支持后端 API 触发“下载平台匹配发布资产 -> staging -> helper 进程切换二进制 -> 重启”。
   - Unix/macOS 下已进一步支持监听套接字继承，重启路径从“停后再起”升级为“先起新进程，再停旧进程”。
   - 本次继续补齐：helper 不再只等 ready 文件，而是会进一步轮询 `/healthz`，并校验响应 PID 必须等于新子进程 PID，确认新进程真正对外可服务后才切旧进程。
   - 同时增加 `drain` 模式：旧进程在收到 handoff drain 标记后，会先进入拒绝新请求状态，再等待真正退出。
   - 本次继续补齐：进程内已统计活跃 HTTP 请求数，退出前会先等待活跃请求归零或超时，再执行最终 `Shutdown()`。
   - 本次继续补齐：已统计 SSE / WebSocket 活跃连接数，并暴露到 `/healthz` 与 `/api/system/status`；System 页面也已显示这些运行态指标。
   - 本次继续补齐：draining 状态下，新的 SSE / WebSocket 连接会直接拒绝；System 页面会自动轮询运行态并提示“服务切换中”。
   - 本次继续补齐：聊天页已能识别 SSE / WebSocket shutdown-draining 信号，并在服务切换期间提示用户稍后重连。
   - 本次继续补齐：新进程启动后会扫描数据库里“用户消息已入库但 assistant 尚未回复”的会话，按原会话顺序自动重新入队执行，避免因重启导致队列内未完成消息永久丢失。
   - 本次继续补齐：聊天页在服务恢复后，会对仍处于 `running` 的当前会话开启被动 SSE 订阅与兜底轮询；如果上次中断时用户消息已入库但回复尚未完成，前端会把该条用户消息标记为处理中，并实时接收恢复后的 `assistant.start / delta / done` 事件。
   - 本次继续补齐：`assistant.start` 事件已显式携带 `user_message_id` 与 `resume` 标志；聊天页可直接知道“当前恢复的是哪条用户消息”，不再完全依赖本地消息配对推断。
   - 本次继续补齐：事件总线已增加短时历史缓冲；SSE / WebSocket 订阅端支持按 `since_id` 回放当前会话最近错过的事件，因此恢复期或被动 SSE 短暂断线后，能补回最近遗漏的流式 token，而不是只能从后续片段继续看。
   - 本次继续补齐：会话恢复过程已显式区分为 `recovering` 状态；系统总览接口与页面会显示 `recovering_sessions` / `running_sessions`，聊天页也会根据 `recovering` 给出更明确的恢复提示，而不再只靠 `running` 推断。
   - 本次继续补齐：会话状态变化已通过 `session.status` 事件实时推送，聊天页会直接更新会话的 `recovering / running / idle` 状态，不再完全依赖轮询才能看到状态切换。
   - 本次继续补齐：聊天页的被动恢复通道已增加 WebSocket fallback。当 SSE 恢复订阅失败或中断时，前端会自动改走 `/ws`，并继续携带 `since_id` 回放最近遗漏事件。
   - 本次继续补齐：TUI 也开始消费 `session.status` 事件，终端态会同步刷新会话状态并提示“恢复中 / 执行中 / 已空闲”。
   - System 页面已提供“下载并重启更新”“重启服务”入口。
   - 代码位置：`cli/runtime_control.go`、`cli/updater/updater.go`、`cli/api/server.go`、`cli/main.go`、`cli/engine/service.go`、`web/src/views/SystemView.vue`、`web/src/views/ChatView.vue`

### 部分实现

1. 自动更新已提升到“Web/后端可执行更新”，但仍未达到 PRD 所写的零中断接管。
   - 已有：GitHub Release 检查、平台资产选择、校验、解包、helper 进程切换二进制、监听继承、旧进程 drain、Web 端触发更新安装与重启。
   - 未有：Windows 下同等级监听继承、长连接无缝续传、当前对话真正零感知不中断。
   - 代码位置：`cli/api/server.go`、`cli/command_cli.go`

2. 优雅重启已提升到“helper 协调 + 监听继承 + handoff 协调”级别，但还不是“会话无损迁移”级别。
   - 当前已支持主进程发起、helper 协调重启。
   - Unix/macOS 下已支持监听继承，新进程会先完成 handoff ready 标记，再通过 `/healthz` + PID 匹配确认可服务后关闭旧进程。
   - 旧进程在切换期间已支持 `drain`：`/healthz` 返回 `503`，普通请求返回 `503 + Retry-After`，避免继续承接新流量。
   - 已增加活跃请求计数与短暂 drain 等待；新进程启动后也会自动恢复数据库中尚未得到 assistant 回复的用户消息，避免重启把内存队列中的待处理消息丢掉。
   - 当前已做到“恢复后自动继续执行 + 前端重新订阅并继续展示后续流 + 短时事件回放补片段”；但仍未实现连接级别的完整迁移、旧 SSE/WebSocket 原连接无缝续接、以及已经流式输出到一半的响应从精确断点继续推送。
   - PRD 中“处理中请求完全无中断、当前对话完全无感知”尚未实现。
   - 代码位置：`cli/runtime_control.go`、`cli/command_cli.go`、`cli/main.go`、`cli/service_manager.go`

3. A2A 已补到“可委托 + 可跟踪 + 可回调 + 可鉴权”的完整主链路，但外部生态级互操作仍需更大范围验证。
   - 已有：`/a2a` HTTP 入口、`A2AClient`、Agent Card、手动注册节点、UDP 组播发现、注册中心同步。
   - 本次补齐：标准化任务状态结构、`accepted/running/partial/completed/failed` 生命周期、`/api/a2a/tasks` 状态查询、回调式状态同步、异步 dispatch、本地 Bearer 鉴权、内建 registry `/register` `/peers` 端点、`peer_url / peer_name / capability` 自动解析，以及协议/API 级自动化测试。
   - 仍未完全证实：跨不同厂商/不同语言 A2A 实现的大规模互操作验收。
   - 代码位置：`cli/protocol/a2a.go`、`cli/protocol/discovery.go`、`cli/api/server.go`

4. 多模态链路已补到“云端优先 + 本地回退”的可用闭环，但仍不是“绝对只允许单一路径”的极端实现。
   - 已有：图片、PDF、Office、压缩包、STT/TTS、图表/HTML、图片生成。
   - 已补齐：上传链路支持可配置病毒扫描 API；非图片文档分析优先走云端 `Responses` 文件输入，失败时回退本地提取器，避免能力直接失效。
   - 仍保留的现实取舍：当云端文件解析不可用或返回错误时，系统仍会回退本地提取，以保证功能连续性。
   - 代码位置：`cli/api/capability_api.go`、`cli/api/server.go`


### 尚未证实或仍需补齐

1. `agentskills.io` 的“严格兼容”仍需外部真实 Skill 包做交叉验证。
   - 当前代码已具备标准 Skill 包兼容层，不再只是本地 `SKILL.md`；并已补齐多行 YAML 列表、块标量、`prompt_file`/`instructions_file` 别名等常见 Skill 包元数据写法。
   - 当前已补齐社区常见 zip 打包形态兼容：单层根目录包在安装时会自动拍平，不再因为 manifest 被包在顶层子目录中而安装失败。
   - 当前仓库已增加本地 zip Skill 包验收样例，覆盖 manifest 别名、资产文件与依赖安装链路。
   - 但“可直接加载社区 28,000+ Skill”仍属于目标表述，当前仓库内仍没有对应规模的外部生态交叉验证。

2. PRD 中关于 MCP 客户端提到的 `mcp-go SDK` 已部分对齐到真实代码。
   - 当前 `stdio` 与标准 `http` Streamable/SSE MCP 客户端链路已优先复用官方 `github.com/modelcontextprotocol/go-sdk/mcp`。
   - 为兼容旧式非 Streamable HTTP JSON-RPC 服务，当前仍保留自研 HTTP JSON-RPC 回退层；因此准确口径应为“SDK-first + legacy fallback”，而不是“所有 HTTP 实现都完全由 SDK 承担”。

3. Skill 驱动的 MCP Server 生态已明显增强，但真实第三方包覆盖范围仍需继续扩大。
   - 当前已支持 builtin + installed Skill 中 `tools/*.json|yaml|yml` 的 MCP Server 自动发现。
   - 当前已补齐 Skill 工具配置中的相对路径解析、`${SKILL_DIR}` / `${TOOLS_DIR}` / 环境变量模板展开，以及 builtin MCP Skill 样例包。
   - 但真实社区 Skill 的 MCP 工具文件格式仍可能继续出现更多变体，仍需持续补样例与交叉验证。

## 本次补缺

本次已补齐的真实实现缺口：

1. Skill 依赖版本约束解析与选择。
2. 远端市场中按约束选择最高满足版本。
3. 对应自动化测试覆盖。
4. 平台发布资产选择、下载解包、checksum 校验。
5. 后端/API 触发的受控重启与更新安装。
6. 聊天页显式切换为 `TDesign ChatSender` 输入层，并用 `TDesign ChatMessage` 承载引用预览。
7. 会话发送门槛与后端能力对齐：前端不再强制“先选会话再发送”，缺失会话时走现有自动建会话流程。
8. 重启 helper 增加健康探针确认：校验 `/healthz` 返回的新 PID 后再切父进程。
9. 旧进程新增 drain 模式：切换期间拒绝新请求，并通过 `/healthz=503` 暴露非就绪状态。
10. 增加活跃 HTTP 请求计数，并在退出前等待请求归零或超时。
11. 增加 SSE / WebSocket 活跃连接计数，并接入系统状态页展示。
12. draining 状态下拒绝新的 SSE / WebSocket 连接，系统页自动轮询并展示切换提示。
13. 聊天页识别 draining/shutdown 信号，并在切换期间禁发消息、提示重连；恢复后会自动刷新状态、重建聊天组件实例，并提示“可继续对话”。
14. 应用全局壳子已增加 draining 告警、自动轮询与恢复成功提示，因此用户在任意页面都能看到服务切换状态与恢复结果。
15. 新进程启动后自动恢复未完成助手任务：扫描每个会话末尾尚未匹配 assistant 回复的用户消息，并按原顺序重新入队执行。
16. 对应自动化测试已覆盖：未回复消息识别、`auto_approve` 元数据持久化、启动恢复入队。
17. 聊天页增加恢复态 UX：恢复提示、当前会话 `running` 状态锁定输入、被动 SSE 补订阅、恢复期兜底轮询、未完成用户消息 pending 标记。
18. 事件总线增加会话级短时回放：SSE / WebSocket 可按 `since_id` 补发最近错过的 `assistant.delta` 等事件。
19. 会话状态增加 `recovering` 语义，系统状态页同步展示恢复中会话与运行中会话数量。
20. 多模态文件解析新增 Office 与压缩包覆盖：补齐 `.xlsx` / `.pptx` / `.zip` / `.tar.gz` / `.tgz` 提取能力，并加入云端图片生成输出。
21. 非图片文档分析已改为云端优先：优先走 OpenAI `Responses` 文件输入解析，失败时自动回退本地提取器。
22. 上传链路已支持可配置病毒扫描 API，启用后会在落库前拦截受感染文件。
23. 引用消息与草稿引用预览已切换为 `TDesign ChatList` 语义渲染，聊天页显式覆盖 `Chatbot + ChatEngine + ChatList + ChatSender + ChatMessage + ChatActionbar + ChatContent + ChatMarkdown + ChatThinking + ChatLoading + Attachments`。
24. Skill 解析器已补齐多行 YAML frontmatter 兼容：支持块标量、缩进列表、以及 `prompt_file` / `instructions_file` 等 manifest 别名。
25. MCP `stdio` 客户端已切换为官方 `mcp-go SDK`，并通过现有 MCP 管理层继续向上提供 `tools/list` 与 `tools/call` 能力。
26. MCP `http` 客户端已切到 SDK-first：标准 Streamable/SSE 服务优先走官方 `mcp-go SDK`，旧式 HTTP JSON-RPC 服务自动回退到兼容实现。
27. Skill 市场已增加第三方风格 zip Skill 包验收测试，覆盖 `prompt_file` 别名、打包资产与依赖安装链路。
28. A2A 任务委托已改为异步主链路：`dispatch` 立即返回任务记录，后台执行后通过状态表推进 `queued -> running -> completed/failed`。
29. A2A 已新增 `/api/a2a/tasks` 与 `/api/a2a/tasks/{id}/status`，支持本地查询与远端回调式状态同步。
30. A2A 协议已补充可选 Bearer 鉴权、 richer Agent Card 字段，以及协议/API 自动化测试。
31. A2A 已补充内建 registry 端点 `/register` 与 `/peers`，可直接作为其他 XClaw 节点的 registry sync 目标。
32. Skill 驱动的 MCP Server 自动发现已覆盖 builtin + installed 目录，不再只限于 installed。
33. Skill 驱动的 MCP Server 已支持相对路径、`${SKILL_DIR}` / `${TOOLS_DIR}` / 环境变量模板展开，更接近真实第三方 Skill 包写法。
34. 已新增 builtin MCP Skill 样例：`filesystem-mcp`、`fetch-mcp`、`github-mcp`，并补齐对应自动发现测试。
35. `/api/a2a/peers` 已统一合并 manual + discovery + registry 视图，并返回可直接 dispatch 的标准化 `url`，避免把 `/a2a` endpoint 再错误拼成 `/a2a/a2a`。
36. `A2A dispatch` 已支持 `peer_name` 与 `capability` 自动选路；`/api/skills/reload` 会同步刷新 Skill 派生的 MCP Server 列表，并返回当前 `mcp_count`。
37. Skill 市场安装链路已兼容第三方 zip 包的单层根目录结构；`/api/skills/install` 现有回归测试已覆盖“社区风格 zip Skill -> 自动注册 MCP Server”整链路。
38. `A2AClient` 已兼容同时接受 base URL 与标准 `/a2a` endpoint 输入，协议测试已覆盖真实 HTTP Server + callback 状态回传，避免跨实现时再次出现 `/a2a/a2a` 路径错误。
39. `A2AClient` 已兼容多种常见返回形态：直接 `A2AResult`、`{"result":{...}}` 包装结果、以及 `202 accepted/queued` 异步确认载荷。
40. A2A callback 状态入参已兼容 `state` 与 `status` 两种字段名，降低不同实现之间的回调对接成本。
41. A2A callback 现已兼容 `task: { id, status/state }` 这类嵌套 envelope，并可在只提供 `result.status` 时自动推断本地任务状态。

## 建议的文档口径

为了避免继续出现“文档已经写成已完成，但代码只是部分完成”的问题，后续文档建议统一使用三类状态：

- 已实现
- 部分实现
- 规划中

在没有完成协议互操作验收、无损重启接管、真实第三方生态兼容验证前，不应继续使用“完整支持”“完全实现”“可直接兼容社区全部生态”这类确定性表述。
