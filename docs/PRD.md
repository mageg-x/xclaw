### 1. 产品愿景

打造一个 **“一人即公司”** 的万能 AI Agent 系统：以一份**单一、无依赖、轻量级的二进制可执行文件**交付给用户。用户从 GitHub Releases 页面下载对应平台的文件，直接运行，打开浏览器访问 `http://localhost:{port}`（可选开启 HTTPS），即可拥有一个**全功能的 AI Agent 运行时环境**。

这个 Agent 本身是一个**无角色的通用执行引擎**。它不预设自己是程序员还是营销专家，不内置任何特定领域的知识或行为模式。它的全部专业能力，均来自用户在管理界面中用**自然语言实时定义的角色描述**，以及为其选择的**工具集**和**云端大模型 API**。

用户可以用它来组建一支虚拟的 AI 团队：创建一个“项目经理”Agent，再创建几个“全栈工程师”、“测试工程师”、“设计师”Agent，将它们拉入同一个工作组。然后只需下达一个高层级目标，整个团队就会自动分解任务、分配工作、并行执行、相互检查、修正错误，直至完成交付。

系统完全基于云端 API 进行推理，不捆绑任何本地模型权重，不要求用户具备 GPU，不需要安装任何 AI 运行时框架。整个二进制文件保持体积小巧、启动迅速、资源占用低，同时依然具备动态角色塑造、多 Agent 协作、工具生态、自主规划、技能市场、跨团队协议、长期记忆进化、人机协同审批、多模态交互等全部高级能力。

所有用户数据（配置、对话历史、记忆文件、Agent 工作空间）均存储在用户本地设备的 **XClaw 根目录** 下：Windows 为 `%APPDATA%\xclaw\`，macOS 为 `~/Library/Application Support/xclaw/`，Linux 为 `~/.xclaw/`。用户对自己的数据拥有完全的主权和控制权。没有强制的云同步，没有隐藏的数据上报，系统可以在完全离线的环境下启动和运行（仅需 LLM API 网络连接）。


### 2. 核心功能

#### 2.1 单二进制全栈交付

这是整个产品最核心的交付形态特征。用户不需要理解任何技术栈，不需要安装任何前置依赖，不需要配置复杂的运行环境。整个系统的所有组件——前端界面、后端 API 服务、Agent 运行时引擎、LLM 网关、数据库——全部编译进一个单独的二进制可执行文件中。

**零依赖安装体验**：
用户从 GitHub Releases 页面根据自己的操作系统和 CPU 架构下载对应的文件。在 macOS 或 Linux 上执行 `chmod +x agent && ./agent`，在 Windows 上双击运行或通过命令行启动。程序默认在本地随机或指定端口上启动 HTTP 服务（可选 TLS），并在默认浏览器中打开管理界面。整个过程从下载到开始使用，不超过一分钟。

**内嵌 Web 管理界面**：
程序内置了一个完整的、功能丰富的 Web 管理面板，使用 Vue 3 构建。这个界面包含了 Agent 实例管理、团队编排、实时对话、任务监控、Token 消耗统计、技能市场浏览、系统设置等所有功能模块。用户通过浏览器完成所有操作，不需要接触任何配置文件或命令行（高级用户也可以通过配置文件进行设置）。

前端构建产物（HTML、CSS、JS、字体、图标等静态文件）在 Go 编译时通过 `//go:embed` 指令嵌入到二进制文件中。程序启动后，由内置的 HTTP 服务器直接将这些静态资源返回给浏览器，不依赖任何外部 CDN 或 Web 服务器。

**首次运行向导（Onboarding Wizard）**：
借鉴 OpenClaw 的 `openclaw onboard` 引导式体验设计，新用户首次启动程序后，浏览器会自动打开一个引导页面。向导会以对话式或分步表单的方式，带领用户完成以下初始化步骤：
1. **设置管理员密码**：为管理面板设置访问密码，保护本地 Agent 环境不被未授权访问。
2. **配置 LLM API**：用户选择要使用的模型提供商（OpenAI、Anthropic、DeepSeek、Moonshot 等），填入对应的 API Key，并选择一个默认模型。系统会自动验证 API Key 的有效性。
3. **创建工作区根目录**：确认 Agent 工作空间和数据的存储位置。默认使用平台标准目录：Windows `%APPDATA%\xclaw\`，macOS `~/Library/Application Support/xclaw/`，Linux `~/.xclaw/`。
4. **创建第一个 Agent**：引导用户用自然语言描述第一个 Agent 的角色（例如“你是一个通用助手，可以帮助我处理各种日常任务”），完成创建后立即进入对话界面开始使用。
5. **可选配置**：是否开启匿名使用统计、是否检查预发布版本更新等。

**跨平台构建与分发**：
利用 Go 语言强大的交叉编译能力，在 CI/CD 流水线中通过 `GOOS` 和 `GOARCH` 环境变量，一次性编译输出以下目标平台的二进制文件：
- macOS: `darwin/amd64`（Intel 芯片）、`darwin/arm64`（Apple Silicon 芯片）
- Linux: `linux/amd64`、`linux/arm64`
- Windows: `windows/amd64`

每个平台的二进制文件独立打包，文件名包含版本号、操作系统和架构信息（例如 `agent_v1.0.0_darwin_arm64.tar.gz`）。配合 GoReleaser 等发布工具，可以自动生成 Release Notes、校验和文件，并推送至 GitHub Releases 页面。用户按需下载，无需任何额外安装步骤。

**自动更新机制**：
程序内置了自我更新能力。每次启动时，会在后台异步检查 GitHub Releases 是否有新版本。当检测到更新后，在 Web 界面顶部显示更新提示条，展示新版本号、更新内容和文件大小。用户点击“更新”按钮后，程序自动下载新版本二进制文件，验证校验和，替换当前运行的文件，然后执行优雅重启（等待当前请求处理完毕后，新进程接管端口，旧进程退出）。

**后台服务模式**：
支持通过 `--daemon` 标志以守护进程（后台服务）方式运行。程序会自动注册为操作系统级别的服务：
- Linux: 创建 systemd unit 文件
- macOS: 创建 launchd plist 文件
- Windows: 注册为 Windows Service
设置为后台服务后，程序随系统启动自动运行，用户随时通过浏览器访问管理面板。


#### 2.2 Agent 工作空间体系（源于 OpenClaw）

这是从 OpenClaw 项目中汲取的核心设计理念之一：把 Agent 的“人设”、记忆、配置全部以文件系统上的 Markdown 文件形式进行管理。这种设计的最大优势在于**透明性、可编辑性、可版本控制性**。用户可以直接用任何文本编辑器打开这些文件查看和修改，也可以将它们纳入 Git 仓库进行版本管理和团队共享。

**文件化角色配置**：
每个 Agent 实例在其独立的工作空间目录下，通过一组约定好的 Markdown 文件来定义其完整的“人格”和行为准则。这些文件在 Agent 会话启动时会被自动加载，作为 System Prompt 的一部分注入到 LLM 的上下文中。

- **`AGENTS.md`** —— 核心操作指令与行为准则文件。这是 Agent 最重要的配置文件，定义了 Agent 应当如何工作、遵循什么流程、有什么限制。例如，一个“全栈工程师”Agent 的 AGENTS.md 会写明：在写代码之前先检查现有项目结构、必须写单元测试、提交前运行 lint 等。Agent 每次会话启动时都会首先读取这个文件。借鉴 OpenClaw 的设计，AGENTS.md 被定位为**地图而非手册**，建议控制在约 100 行以内，仅做索引和指引，避免过度挤占 LLM 有限的上下文窗口。更详细的规范文档放在工作空间的子目录中，按需加载。

- **`SOUL.md`** —— 人设、语气、表达风格与道德边界定义文件。这里描述 Agent 的“性格”和“说话方式”。例如：“你是一个严谨但友善的技术专家，喜欢用简洁的代码示例来说明问题。你从不使用过于花哨的营销语言，也不会对不确定的事情给出武断的结论。” 这个文件确保了即使任务内容变化，Agent 的回复风格始终保持一致。

- **`IDENTITY.md`** —— 名称、Emoji 图标与视觉标识文件。定义 Agent 的显示名称（如“张三-全栈工程师”）、代表 Emoji（如 👨‍💻）、头像设置等。这些信息在 Web 界面的对话列表、团队成员展示等位置使用，帮助用户快速识别不同的 Agent。

- **`USER.md`** —— 服务对象画像与称呼偏好文件。Agent 会通过长期交互逐渐学习并填充这个文件，内容包括用户的称呼方式、编码风格偏好、常用技术栈、沟通习惯、甚至个人兴趣等。Agent 在对话中会参考这个文件，提供更加个性化的服务。例如，Agent 知道用户喜欢使用中文注释，就永远不会生成纯英文注释的代码。

- **`TOOLS.md`** —— 工具使用说明与本地约定文件。描述 Agent 可使用的工具列表、每个工具的用途和使用场景、以及特定于本工作空间的工具使用约定。例如，文件写入工具的默认编码格式、Shell 执行工具的默认工作目录等。

- **`MEMORY.md`** —— 经过整理的长期记忆沉淀文件。Agent 会主动将跨会话的重要信息、项目关键决策、已验证有效的操作流程等，以结构化的格式写入这个文件。这个文件可以看作 Agent 的“外挂大脑”，帮助它在跨天、跨周的任务中保持连续性。

- **`memory/YYYY-MM-DD.md`** —— 每日按日期归档的 Markdown 记忆日志。Agent 会在每天的工作中，将重要的交互信息、执行结果、遇到的问题和解决方案等，自动记录到对应日期的文件中。在每次新会话启动时，Agent 会自动读取当天和昨天（可配置天数）的记忆文件内容，将其注入到初始上下文中，从而自然地实现跨会话的知识连续性。这种日期分片的设计使得记忆文件不会无限增大，旧文件可以按需归档或删除。

**Markdown 持久化记忆系统**：
Agent 的记忆管理是保障长期可用性的核心。传统的做法是依赖向量数据库 + 嵌入检索，这在技术上虽然成熟，但对用户来说是一个黑盒，难以检查和编辑。本系统采用的 Markdown 文件记忆方案，则提供了极大的透明性和可控性。

Agent 在对话过程中，会根据内容的重要性自动触发记忆写入操作。例如：
- 用户明确说“记住，我的项目使用 Go 1.21 版本，数据库是 PostgreSQL”
- Agent 完成了一个复杂任务，总结出“在部署前必须先运行 `make lint` 和 `make test`”
- 用户纠正了 Agent 的一个错误做法，Agent 将教训写入记忆

这些记忆以可读的 Markdown 格式写入对应的日期文件中，并用清晰的标题、列表和代码块组织。用户随时可以用文本编辑器打开查看，也可以手动修改或删除不想要的记忆。记忆文件与角色配置文件共同存储在 Agent 专属的工作区目录中，整个目录可以打包备份、迁移到新设备，也可以直接纳入 Git 仓库实现团队共享和版本追溯。

**工作空间严格隔离**：
每个 Agent 实例在文件系统上拥有一个完全独立的目录（例如 Linux 为 `~/.xclaw/data/workspaces/{agent-id}/`，Windows/macOS 为对应 XClaw 根目录下的 `data/workspaces/{agent-id}/`）。这个目录包含了上述所有配置文件、对话历史、记忆日志、以及 Agent 在沙箱中读写的任何文件。

Agent 之间的文件操作绝不交叉。Agent A 只能在自己的工作区内读写文件，无法访问 Agent B 的工作区。这种文件系统级别的物理隔离，从根本上消除了一个 Agent 的误操作（如错误的 `rm -rf`）或幻觉行为影响到其他 Agent 的可能性。

系统支持全局默认工作空间模板配置。用户可以为“全栈工程师”这类常见角色创建模板，新建同类型 Agent 时直接复制模板配置，只需修改名称等基本信息即可。同时，每个 Agent 的工作空间都可以随时进行个性化覆盖，灵活性不受模板限制。


#### 2.3 跨平台原生安全沙箱（零 Docker 依赖）

安全沙箱是保障 Agent 安全运行的基础。当 Agent 被赋予执行 Shell 命令、读写文件、访问网络等能力时，必须将这些能力限制在一个受控的边界内，防止误操作或恶意指令对用户主机造成损害。

本系统的核心设计原则是：**完全不依赖 Docker 或任何容器运行时**。在用户的设备上强制要求安装 Docker 是一种负担，在 Windows 和 macOS 上尤其突出。因此，本系统直接利用各操作系统的原生安全机制来实现进程级别的隔离。

**操作系统原生隔离机制（Native Sandbox）**：
Go 运行时通过系统调用直接设置子进程的隔离属性。不同平台使用不同的内核机制：

- **Linux**：
  - 使用 `chroot` 将工具进程的根文件系统限制在 Agent 的工作区目录范围内，工具进程无法访问工作区以外的任何文件。
  - 使用 `seccomp`（Secure Computing Mode）配置系统调用过滤器，只允许一组安全的白名单系统调用通过。诸如 `reboot`、`mount`、`kexec_load` 等危险系统调用会被内核直接拦截。
  - 使用 `cgroups`（Control Groups）限制工具进程的 CPU 使用率和内存占用上限，防止某个工具的失控消耗大量系统资源。
  - 可选使用非特权用户命名空间（User Namespace），让工具进程在容器中以 root 运行但在主机上实际是非特权用户，增加额外隔离层。

- **macOS**：
  - 使用 App Sandbox 能力，通过 `sandbox-exec` 命令或继承父进程的沙箱配置，限制工具进程的文件系统访问范围、网络连接权限、以及与其他进程的通信能力。
  - 文件访问限制在 Agent 工作区目录内，任何对系统目录或用户其他目录的读写都会被内核拒绝。

- **Windows**：
  - 使用 Job Objects 将工具进程纳入一个受控的作业对象中，限制其 CPU 和内存使用。
  - 使用受限令牌（Restricted Tokens）来降低工具进程的权限级别，移除 `SeShutdownPrivilege` 等危险特权。

所有这些机制完全由 Go 代码通过标准库的 `os/exec` 结合平台特定的 `syscall` 或 `golang.org/x/sys` 包实现，不需要调用任何外部可执行文件。二进制文件保持自包含，沙箱能力随程序一起分发，跨平台一致。

**可配置隔离级别**：
用户可以根据自己的安全需求和信任程度，为每个 Agent 实例独立配置沙箱的严格程度：

- **`off`** —— 沙箱完全关闭。工具进程直接在主机环境中执行，拥有当前用户的所有权限。这种模式适用于完全信任的单用户个人开发环境，性能开销最小。但需要用户明确知晓风险并主动选择此选项。

- **`native`** —— **默认模式**，也是推荐模式。对所有工具调用启用上述操作系统原生隔离机制。文件系统访问被严格限制在 Agent 工作区目录内。系统调用受到限制。这个模式提供了良好的安全边界，同时保持了较低的性能开销和零外部依赖。

- **`custom`** —— 高级用户自定义模式。用户可以指定自己系统上安装的第三方沙箱命令（如 `firejail`、`bubblewrap`、`apparmor` 等），由这些外部程序来提供隔离。程序会将工具调用的命令行参数交给用户指定的沙箱程序来执行。这个模式为有特殊安全需求的用户提供了灵活性。

**工作区访问控制**：
即使在沙箱开启的情况下，用户还可以进一步控制工具进程对 Agent 自身工作区目录的访问权限：

- **`none`** —— 工具进程完全无法访问任何持久化文件。即使 Agent 拥有文件读写工具，工具进程也无法接触到 Agent 工作区。这种模式适用于纯信息查询类的 Agent，例如只调用搜索 API 的助理。

- **`ro`（只读）** —— 以只读方式挂载 Agent 工作区。工具进程可以读取工作区内的文件内容，但任何写入、修改、删除操作都会被操作系统级权限拦截。适用于需要参考配置或记忆但不产生新文件的 Agent。

- **`rw`（读写）** —— 以读写方式挂载，赋予完整访问权限。这是开发类 Agent 常用的模式，因为需要读写代码文件。用户应按需开放，仅在 Agent 职责明确需要文件写入时才使用此级别。

**可选容器增强（非内置，不捆绑）**：
虽然系统默认不依赖 Docker，但设计上预留了容器集成的接口。如果用户环境中已自行安装 Docker 或 Podman，并且愿意承担容器运行的额外资源开销，可以在配置中将沙箱模式切换为 `container`。

启用容器模式后，工具调用会在一个临时创建的 Docker/Podman 容器中执行。容器模式的优势在于提供了一层额外的、业界成熟的隔离层，文件系统、网络、进程空间完全独立。劣势在于需要用户提前安装容器运行时，且启动容器有一定延迟。

**强调：本程序本身不捆绑任何容器镜像，不要求用户安装 Docker，不内置任何容器运行时代码。** 容器模式仅在用户主动选择、且系统中检测到可用容器运行时的情况下才会生效。这完全是一个增值的可选项，而非基线要求。

**审计与监控**：
所有工具调用的完整生命周期都被详细记录在审计日志中，包括：
- 调用时间戳
- 发起调用的 Agent 身份
- 调用的具体命令和参数
- 工具进程的 PID
- 进程的执行时长（从启动到退出）
- 标准输出（stdout）内容
- 标准错误（stderr）内容
- 进程退出码
- 沙箱模式与配置
- 是否触发安全限制

审计日志存储在 SQLite 数据库中，用户可以在 Web 面板的工具调用历史页面中实时查看和搜索。每条记录都可展开查看详细信息。这让用户可以回溯 Agent 的任何操作，实现完全的透明和可审计。


#### 2.4 动态角色塑造

这是本系统区别于传统“为每个任务写一个 Agent”模式的核心能力。传统方案中，如果要让 Agent 承担“项目经理”的职责，开发者需要精心设计一套 Prompt、一个特定的工具链、可能还需要微调模型。如果要切换到“全栈工程师”，又需要另一套完全不同的配置。

本系统的核心设计是：Agent 引擎本身**完全无角色**。它只是一个执行“思考→行动→观察”循环的通用推理与工具调用引擎。它的角色、专业领域、行为风格，全部由用户在创建 Agent 时用日常自然语言的一句话（或一段话）来定义，并在运行时动态注入到 LLM 请求中。

**自然语言零门槛定义角色**：
用户无需任何 Prompt Engineering 知识，无需理解“System Prompt”、“Few-shot”、“Chain of Thought”这些概念。在创建 Agent 的界面上，只有一个大的文本输入框，用户可以像跟人说话一样写下对 Agent 角色的描述。

例如：
- “你是一个资深全栈工程师，有 10 年 Go 和 React 开发经验。写代码必须带详细的中文注释，遵循 Google Go Style Guide。在提交代码前，先自己运行一遍测试确保没有问题。遇到不确定的技术方案时，列出几个选项并说明各自的优缺点，让我来决策。”
- “你是一个项目经理，负责整个项目的任务分解、进度跟踪和团队协调。你的成员包括两个全栈工程师和一个 UI 设计师。你会先理解我的需求，然后拆分为可执行的子任务，分配给合适的成员，并定期向我汇报进度。当遇到阻塞时，你会主动提出解决方案。”
- “你是一个严谨的财务分析师。我会上传公司的财报 PDF，你需要提取关键数据，计算财务指标，并用中文生成一份结构化的分析报告，包括同比分析、风险评估和改进建议。”

系统接收这段自然语言描述后，会将其作为 System Instruction 的一部分，在每次调用 LLM 时固定在 Prompt 的最前面。现代 LLM（GPT-4、Claude 3.5 等）对自然语言指令的理解已经极其强大，用户的口语化描述完全足以让模型准确理解并扮演对应角色。不需要复杂的模板、标签或结构化 Prompt。

**工具绑定与权限控制**：
创建 Agent 时，用户会看到一个可用工具列表（从工具市场中已安装的工具）。用户通过勾选的方式，为当前这个 Agent 实例授权可调用的工具。

例如，一个“全栈工程师”Agent 可能会被授予：`read_file`、`write_file`、`execute_shell`、`search_code` 等工具。而一个“数据分析师”Agent 可能只有 `read_file`、`sql_query`、`generate_chart` 等工具的权限，没有 Shell 执行权限。

工具权限与角色自动关联。在将工具 Schema 注入 LLM 请求时，系统只会列出当前 Agent 已授权的工具，Agent 在推理阶段就不会“看到”那些无权使用的工具。这既是一种安全措施，也是一种 Token 优化——不需要的工具 Schema 不会浪费上下文。

**多模型独立配置**：
每个 Agent 实例可以独立配置使用的底层大模型和相关参数。这意味着用户可以根据任务的重要性、复杂度和预算，为不同 Agent 分配不同级别和成本的模型：

- 项目经理 Agent（需要复杂的推理和规划能力）：使用 GPT-4o 或 Claude Opus
- 前端开发 Agent（生成代码，中等复杂度）：使用 Claude Sonnet 或 DeepSeek-V3
- 数据处理 Agent（结构化提取，相对简单）：使用 GPT-4o-mini 或 DeepSeek-V2

每个 Agent 还可以配置独立的模型参数，如：
- `temperature`：控制回复的随机性。代码生成 Agent 可能设置更低（0.1-0.3），创意文案 Agent 可能设置更高（0.7-0.9）
- `max_tokens`：单次回复的最大 Token 数
- `top_p`：Nucleus 采样参数

每个 Agent 的 API Key 也是独立管理的。模型凭证（API Key）与 Agent 配置绑定，加密存储在本地数据库中。系统支持为不同的 Agent 使用不同提供商的 API Key，实现成本和能力的精细化管理与隔离计费。


#### 2.5 泳道队列并发调度（源于 OpenClaw）

当多个用户同时与不同的 Agent 交互，或者同一 Agent 同时收到多个请求时，并发控制就变得至关重要。LLM 调用本身可能较慢（几秒到几十秒），如果处理不当，会导致消息乱序、状态覆盖等难以调试的问题。

本系统借鉴 OpenClaw 的 Lane Queue（泳道队列）架构，在 Agent 运行时层面实现细粒度的并发调度，从设计根源上保证消息的顺序性和状态的一致性。

**Lane-aware FIFO 队列机制**：
Lane（泳道）是一个逻辑上的消息队列。每个 Agent 会话（Session）自动分配一个独立的 Session Lane。这个 Lane 是一个严格的 FIFO（先进先出）队列，同一 Session 内的所有消息，必须按到达顺序逐个串行执行。

例如，用户在一个对话中连续发送了三条消息：「写一个登录接口」「加上 JWT 认证」「再补上单元测试」。这三条消息会按照发送顺序进入该会话的 Lane 队列。Agent 运行时依次处理每条消息，确保前一条消息执行完毕、状态更新完成后，才开始处理下一条。这保证了对话的因果一致性——「加上 JWT 认证」一定会在「写一个登录接口」完成后才开始。

**双层队列管控**：
除了会话级别的 Lane，系统还实现了全局级别的 Lane 用于更粗粒度的资源协调：

- **Session Lane（会话泳道）**：每个 Agent 的每个对话会话各自独立。同一会话的消息严格串行，保证对话状态和 Agent 内部记忆的因果一致性。不同 Session 的消息可以并行处理，互不影响。

- **Global Lane（全局泳道）**：跨会话的共享资源通过全局泳道进行公平调度。例如，与 LLM API 的连接池大小是有限的，全局泳道确保不会有一个高频率的 Agent 占用所有连接资源，导致其他 Agent 的请求饥饿。文件句柄、数据库连接等共享资源也通过全局泳道管理。

**可配置并发度**：
不同层级的泳道可以配置不同的并发度，适应不同的负载场景：

- 主会话泳道（用户直接交互的 Agent）默认并发度为 4。这意味着一个 Agent 可以同时处理 4 个不同 Session 的请求。
- 子 Agent 泳道（被 Supervisor Agent 调度的执行 Agent）默认并发度为 8。这些 Agent 通常执行较为单一的任务，并发能力可以更高。
- 未显式配置的泳道默认串行（并发度为 1），安全优先。

用户可以在 Agent 配置文件中调整这些参数，也可以为特定 Agent 设置自定义的并发度。系统在 Web 面板提供泳道状态的可视化展示，包括每个 Lane 的队列长度、当前处理中的消息、平均等待时间等。


#### 2.6 动态规划与自我修正

这是 Agent “智能性”的真正体现。一个只会按固定流程执行的程序不能被称为 AI Agent，真正的 Agent 必须能够根据实时情况调整自己的行为，在遇到挫折时重新规划，在完成任务后自我反思。

**ReAct 推理循环**：
每个 Agent 实例都运行标准的 ReAct（Reasoning and Acting，推理与行动）循环。这是一个“思考 → 行动 → 观察 → 再思考 → ... ”的持续迭代过程：

1. **思考（Reason）**：LLM 接收当前状态（用户目标、已完成的步骤、最新的观察结果），进行分析和推理，输出一个决策——下一步应该做什么？是调用某个工具，还是给出最终回答？
2. **行动（Act）**：如果 LLM 决定调用工具，Agent 运行时就在沙箱中执行该工具调用。
3. **观察（Observe）**：工具执行完毕，Agent 读取执行结果（成功/失败、输出内容、错误信息等）。这个结果被追加到对话上下文中。
4. **循环**：带着新的观察结果，Agent 再次进入“思考”阶段。根据最新信息判断：是继续行动、调整计划、还是任务已完成？

这个循环持续进行，直到 LLM 在“思考”阶段决定不再调用工具，而是直接输出最终结果回复给用户。

**计划自适应重排（Replan）**：
ReAct 循环天然支持计划的动态调整。Agent 不需要在一开始就制定一个完整的、不可更改的计划。它可以在执行过程中根据反馈不断修正自己的策略。

举例说明：用户让 Agent“把这个项目部署到生产环境”。Agent 的初始计划可能是：
1. 运行测试 → 2. 构建 Docker 镜像 → 3. 推送到镜像仓库 → 4. 更新 Kubernetes 部署

但在第 1 步时，测试失败了。Agent 观察到错误信息后，重新“思考”：测试失败意味着代码有问题，部署不能继续。于是自动调整计划为：
1. 分析测试失败原因 → 2. 修复代码 → 3. 重新运行测试 → 4.（如果通过）继续部署

又比如，在第 3 步推送镜像时，发现镜像仓库连接不上。Agent 再次调整：检查网络连接、尝试备用仓库、或报告给用户请求指示。

这种灵活的、基于实时反馈的决策能力，是 ReAct 模式相比 Plan-and-Execute（先制定完计划再执行）模式的最大优势，也是本系统 Agent 能胜任各类不确定、探索性任务的核心原因。

**反思沉淀与知识进化**：
当任务最终完成（无论成功还是失败），Agent 会执行一个“反思”步骤。在这个步骤中，Agent 会回顾整个任务的执行过程，总结出：
- 哪些步骤做对了，值得保留为经验
- 哪些步骤走了弯路，下次可以如何改进
- 本次任务中发现的新的用户偏好或项目约定
- 是否有值得写入长期记忆的重要信息

这些反思结果会自动沉淀为 Markdown 格式的总结，写入 `MEMORY.md` 或当天的日期记忆文件中。随着使用时间的增长，Agent 积累的经验会越来越多，处理同类任务时就会更加高效和准确。这个自我进化的闭环是 Agent 从“能用”走向“好用”的关键。


#### 2.7 Harness Engineering 工程化约束体系

Harness 工程的核心理念是：为 Agent 构建一个“行为操作系统”，用明确的、确定的规则来约束 Agent 的行为边界，而不是靠 Prompt 中的模糊期望。Agent 在规则内可以自由发挥，但绝对不能越界。

**AGENTS.md 规范驱动**：
这是 Harness 理念的核心实践——把项目规范、编码标准、架构约束等“基础设施即代码”。每个项目的 AGENTS.md 文件是这个项目给 Agent 的“入职手册”。

例如，一个 Go 项目的 AGENTS.md 可能包含：
```markdown
# 项目规范
- 本项目使用 Go 1.22，模块路径为 github.com/mycompany/myapp
- 代码必须通过 `golangci-lint run` 检查，零告警
- 所有公开函数必须有 GoDoc 注释
- 数据库变更必须放在 migrations/ 目录下
- 测试覆盖率不低于 80%
- 禁止使用 `panic`，使用 `error` 返回
- 配置文件统一使用 Viper，放在 configs/ 目录
```

AGENTS.md 的设计原则是“地图而非手册”。文件控制在约 100 行，只做高层索引和核心规则指引。更详细的规范（如 API 设计规范、数据库命名规范等）放在项目目录的 `docs/` 或 `.xclaw/rules/` 子目录中，Agent 在需要时通过工具调用按需读取。这样既不会被大量规范文本淹没 LLM 的上下文窗口，又保证了规范信息的完备性。

在每次会话启动时，Agent 自动加载项目的 AGENTS.md。随后的所有工作——无论是写代码还是做设计——都在这些规则框架内进行。AGENTS.md 文件被纳入 Git 仓库管理，团队成员共同维护，保证了规范的一致性和可追溯性。

**自动化验证管道**：
Harness 工程强调“用确定性的机械化检查替代 LLM 的模糊直觉”。LLM 可能会因为上下文压缩、模型版本差异、Prompt 微小变化等原因而产生不一致的输出。但 Lint 工具、类型检查器、单元测试这些确定性工具的输出永远是一致的。

在每次 Agent 完成代码修改后、在执行部署等危险操作前，系统会自动触发验证管道：
1. **Lint 检查**：调用项目配置的 Lint 工具（如 golangci-lint、eslint），检查代码风格和潜在错误
2. **类型检查**：对于强类型语言，运行编译器类型检查（如 `go vet`、`tsc --noEmit`）
3. **单元测试**：运行项目的测试套件
4. **安全扫描**（可选）：调用配置的安全扫描工具检查已知漏洞

如果任何一步验证失败，Agent 会自动收到失败信息，并被要求修正代码。它会进入一个“修正循环”：分析失败原因 → 修改代码 → 重新运行验证 → 直到全部通过。这个过程模拟了 CI/CD 管道的行为，但将拦截时机从“合并后才发现”提前到了“代码生成后立即发现”。

**自我进化闭环**：
Agent 不会在同一个地方跌倒两次。系统设计了两层机制来实现持续改进：

- **从失败中学习**：当 Agent 的一次工具调用失败（如命令执行返回非零退出码、API 返回错误响应），系统会记录失败的完整上下文（调用的命令、参数、环境、错误信息）。Agent 在反思阶段分析失败根因，如果判断这是一个可预防的错误，会将经验总结为一条新规则，追加到 AGENTS.md 或 MEMORY.md 中。例如：“注意：部署前必须先设置 KUBECONFIG 环境变量”、“这个 API 的认证 Token 每 30 分钟过期，需要提前刷新”等。

- **编译确定性脚本**：对于频繁执行的重复性任务序列（如项目初始化 `npm install && npm run build`、部署流程 `docker build && docker push`、数据备份脚本等），Agent 会在多次执行后，将这些经过验证的步骤编译为一个确定性的 Shell 脚本，缓存在工作空间中。后续遇到相同或高度相似的任务时，Agent 不再通过 LLM 推理每一步如何操作，而是直接调用缓存脚本，将 LLM 完全移除出执行路径。这既大幅降低了 Token 消耗，也消除了 LLM 可能产生幻觉或不一致的风险。


#### 2.8 Agent 技能市场

为了保持核心二进制文件的精简，同时赋予系统无限的扩展能力，本系统内置了一个完整的技能市场（Skills Marketplace）功能模块。用户可以像安装手机 App 一样，为自己的 Agent 安装新的能力。

**技能市场的生态模式**：
借鉴 OpenClaw 的 ClawHub 生态设计，技能市场允许第三方开发者（或用户自己）将自定义的工具、技能包、角色模板等打包发布，供社区共享。一个技能包（Skill Package）是一个标准化的打包文件，包含：

- `skill.yaml` —— 技能包的元数据描述文件，包括名称、版本、作者、描述、图标、依赖项等
- `tools/` —— 工具定义文件（如 MCP 工具配置、自定义脚本）
- `agents/` —— 角色模板文件（预置的 AGENTS.md、SOUL.md 等）
- `README.md` —— 技能包使用说明

Skill 格式：严格遵循 agentskills.io 标准规范。这里的“遵循”不是营销表述，而是明确的兼容性要求：XClaw 必须支持标准 Skill 包结构、manifest 元数据、版本约束、依赖关系、安装来源、锁定文件与校验规则；并且要接受真实社区 Skill 中常见的 `SKILL.md` frontmatter 写法，例如多行 YAML 列表、块标量描述、`prompt_file` / `instructions_file` 等字段别名。当前仓库中的本地 `SKILL.md` 体系只作为扩展层存在，不能替代标准 Skill 包兼容层。
工具加载与执行：通过 MCP（Model Context Protocol）协议，以语言无关的方式加载和调用任何工具。这里要求的是 **MCP 客户端实现**，而不仅仅是把 XClaw 自己暴露成一个 MCP Server。Agent 内核需要完成外部 MCP Server 的注册、连接、握手初始化、能力协商、工具列表获取、工具调用、超时取消、健康检查、错误归类、权限审批与审计落盘。实现约束上，`stdio` 与标准 `http` Streamable/SSE MCP 客户端链路优先复用官方 Go `mcp-go SDK`；旧式非 Streamable HTTP JSON-RPC 服务允许保留兼容回退层，只要对上层暴露统一 MCP 客户端能力。
生态目标：完成后既能加载社区的 28,000+ Skill，也能调用社区中用任何语言编写的海量工具；这里的“能加载”是兼容目标，不等同于仓库内置了全部生态样例。验收标准是“第三方 Skill 与 MCP Server 可在不改本地代码的前提下直接安装、注册与运行”。

系统提供官方的技能注册中心（Registry），开发者可以将技能包发布到注册中心。用户也可以在本地管理自己开发或下载的技能包，无需连接外部网络即可安装本地技能包。

**Web 界面内的完整市场体验**：
技能市场完全集成在 Web 管理面板中，用户无需使用命令行或编辑配置文件：

1. **浏览与搜索**：市场页面展示技能包的列表，包含名称、图标、简介、下载量、评分等信息。用户可以按分类（编程、数据分析、设计、办公自动化等）、标签、关键词搜索。
2. **详情查看**：点击任一技能包，进入详情页。展示完整的 README、版本历史、所包含的工具和角色模板说明、用户评价等。
3. **一键安装**：点击“安装”按钮后，系统自动完成：下载技能包文件 → 解压到本地缓存目录（XClaw 根目录下的 `cache/skills/`）→ 注册工具定义到系统中 → 安装角色模板到模板库。整个过程无需重启程序。
4. **管理与更新**：已安装的技能包在“我的技能”页面集中管理。可以查看当前版本、检查更新、回滚到旧版本、卸载等。技能包按来源和版本隔离存储，互不干扰。

**技能运行的安全约束**：
从市场安装的技能包中如果包含可执行脚本或工具，它们的运行严格受系统沙箱机制的限制。安装过程中，用户会看到技能包所请求的权限列表（如文件读写、网络访问、Shell 执行），用户可以选择授予或拒绝某项权限。运行时，所有工具调用均在 Agent 的沙箱环境中执行，不会越权访问系统资源。


#### 2.9 跨团队 Agent 协作（A2A 及其他协议）

当用户创建了多个 Agent 并组建了团队后，Agent 之间需要能够高效、安全地进行通信和协作。更进一步，不同用户在不同设备上部署的 Agent 实例之间，也可能需要相互委托任务。这就需要标准化的 Agent 间通信协议。

**A2A（Agent-to-Agent）协议完整支持**：
A2A 是 Google 力推的开放标准协议，专为 AI Agent 之间的互操作而设计。本系统完全实现 A2A 协议栈，使得任何一个由本系统创建的 Agent，都能够：
- **发现**：在网络中自动发现其他 A2A 兼容的 Agent
- **通信**：建立安全的、经过认证的通信信道
- **委托**：将子任务打包为结构化的 A2A 任务描述，发送给另一个 Agent
- **监控**：跟踪委托任务的执行状态
- **接收结果**：接收结构化的执行结果或中间状态更新

例如，用户本地运行的主 Agent 可以将一个“生成数据可视化图表”的子任务，通过 A2A 协议委托给团队另一个成员电脑上正在运行的、专门配置了数据可视化工具包的 Agent。任务完成后，图表结果通过协议传回，整个过程对用户来说是透明的。

**Agent Card 自动发现机制**：
每个 Agent 实例在启动时，会生成一张符合 A2A 规范的 Agent Card（能力名片），内容包括：
- Agent 的唯一标识符和名称
- Agent 的角色和能力描述
- 支持的通信协议和端点 URL
- 可调用的工具和能力列表
- 支持的任务类型

Agent Card 可以通过以下机制被其他 Agent 发现：
- **局域网广播（mDNS/DNS-SD）**：同一局域网内的 Agent 自动相互发现
- **注册中心**：用户配置的公共或私有注册中心，Agent 启动时自动注册，退出时自动注销
- **手动添加**：用户手动输入另一个 Agent 的地址和端口来建立连接

**多协议兼容**：
除了 A2A 作为 Agent 间的核心协作协议外，系统同时内建支持：
- **MCP（Model Context Protocol）客户端**：用于接入外部工具服务器，无论这些工具是本地运行的服务还是远程 API
- **HTTP RESTful API**：供外部系统通过标准 HTTP 接口调用 Agent，或查询 Agent 状态
- **SSE（Server-Sent Events）**：用于向客户端推送 Agent 的流式响应和实时状态更新
- **WebSocket**：用于双向实时的 Agent 对话交互


#### 2.10 长期记忆进化与个性化

Agent 的真正价值不仅仅在于完成单次任务，更在于它能够随着使用时间的增长，越来越了解用户、越来越懂得用户的偏好、越来越高效。这需要一个强大的长期记忆系统和个性化学习机制。

**基于文件 + 向量的混合记忆架构**：
系统的记忆存储采用双轨制：

- **文件记忆**：前文所述的 Markdown 日期分片文件和 MEMORY.md 文件，作为记忆的“权威源”（Source of Truth）。这些文件是人类可读、可编辑、可 Git 版本管理的。用户随时可以打开查看 Agent “记住了什么”，也可以纠正或删除不想要的记忆。

- **向量记忆**：所有 Markdown 记忆文件的内容，会被异步提取为文本片段，通过指定的 Embedding API（如 OpenAI text-embedding-3-small）生成向量嵌入，存储在本地 SQLite 的向量扩展（如 sqlite-vec）中。当 Agent 需要检索历史记忆时（例如：“用户上次提到的那个数据库配置是什么？”），系统会先将查询转为向量，在向量库中进行语义相似度搜索，找到最相关的记忆片段，再将其注入到 LLM 的上下文中。

这种混合架构结合了两者的优势：文件保证了透明性和可控性，向量保证了在大规模记忆中的快速语义检索能力。

**用户偏好自动学习**：
Agent 会在与用户的长期交互中，持续观察并学习用户的偏好。这些偏好被自动记录和更新在 `USER.md` 文件中。学习的内容包括但不限于：

- **技术偏好**：用户喜欢的编程语言、框架、工具链
- **编码风格**：缩进是用 Tab 还是空格、命名是驼峰还是下划线、注释的语言和详细程度
- **沟通风格**：用户喜欢详细的解释还是简洁的结论、是喜欢先看代码还是先看思路
- **决策模式**：用户在做技术选型时更看重性能、开发效率、还是生态成熟度
- **个人习惯**：用户一般什么时候在线、喜欢被打断还是不被打断、对风险的容忍度

随着 `USER.md` 文件的不断丰富，Agent 在每次会话启动时都会加载这些信息，并在对话中参考它们。一个使用了一个月的 Agent，和一个刚刚创建的 Agent，面对同一个技术问题时，给出的建议方式会完全不同——前者会贴合用户的习惯，后者只能给出通用回答。

**知识库挂载与索引**：
除了 Agent 自身在交互中积累的记忆，用户还可以主动为 Agent 指定外部知识源。系统支持挂载以下类型的知识库：

- **本地文件目录**：指定一个或多个本地目录，Agent 会索引目录中的文件（支持 Markdown、纯文本、代码文件、PDF 等格式）
- **Git 仓库**：指定一个 Git 仓库地址，Agent 会 clone 仓库并索引其中的文档和代码
- **Notion 页面**：通过 Notion API 连接用户的 Notion 工作区，索引指定的页面和数据库
- **网页**：指定 URL 列表，Agent 定期抓取并索引内容

索引过程包括文本提取、分块、向量嵌入，最终存储在向量库中。在对话中，Agent 可以实时检索这些外部知识，作为回答的参考。例如，用户问“我们的 API 文档中，用户认证接口的参数是什么？”，Agent 可以从已索引的 API 文档中精确检索到相关信息。


#### 2.11 人机协同工作流与审批

完全的自动化有时不是最佳选择。在某些关键操作上，人类用户的判断和确认是不可替代的。本系统在自动化工作流中内置了审批节点机制，实现“AI 自动化执行 + 人类关键决策”的协同模式。

**可配置审批节点**：
用户可以为 Agent 定义哪些类型的操作需要经过人工审批。审批规则可以基于多种条件：

- **按工具类型**：例如，“所有 Shell 执行操作需要审批”、“文件删除操作需要审批”
- **按操作影响范围**：例如，“涉及生产环境数据库的操作需要审批”、“删除超过 10 个文件时需要审批”
- **按操作参数**：例如，“执行金额大于 1000 元的支付指令需要审批”
- **按操作目标路径**：例如，“对 `/etc/` 或系统目录的任何写入操作需要审批”

审批规则在 Agent 配置中设置，可以按 Agent 粒度独立配置。当 Agent 在执行过程中触发了一个需要审批的操作时，它会暂停当前任务，向审批系统发起一个审批请求。

**审批请求与交互流程**：
审批请求会通过以下渠道通知用户：

- **Web 界面弹窗**：在管理面板中弹出审批请求卡片，显示操作的详细信息（哪个 Agent、要做什么操作、参数是什么、为什么要做这个操作）。卡片上提供“批准”和“拒绝”按钮，用户还可以附加文字意见。
- **系统通知**：通过操作系统通知机制发送桌面通知（支持 macOS、Windows、Linux）
- **Webhook**：用户可配置 Webhook 将审批请求推送到外部系统（如企业微信、Slack、钉钉等）

用户批准后，Agent 恢复执行，并收到“已批准”的状态反馈，继续后续步骤。用户拒绝后，Agent 收到拒绝信息以及用户可能附加的修改意见，Agent 会重新调整方案或请示用户下一步。

用户还可以在审批时选择“批准并记住此决定”。Agent 会将这个审批决策写入记忆，后续同类操作自动放行，减少重复审批的干扰。

**审批策略模式**：
系统支持多种审批策略，适应不同团队规模和工作模式：

- **单人审批（默认）**：任何一个管理员用户确认即可通过
- **会签**：配置的多个审批人全部同意才能通过（适用于高风险操作）
- **或签**：配置的多个审批人中任意一个同意即可通过
- **自动过期**：审批请求在指定时间后如果无人处理，自动按预设策略拒绝（适用于时间敏感的操作）


#### 2.12 多模态交互能力（100% 云端 API）

Agent 不应局限于纯文本交互。现实世界的数据是多种形态的——截图、设计稿、PDF 文档、语音指令——Agent 需要能够理解和生成这些不同形态的内容。本系统的多模态能力完全通过调用云端 API 实现，不捆绑任何本地模型。

**输入多模态——让 Agent 看懂世界**：
用户可以在对话中上传多种类型的文件作为任务输入的一部分：

- **图片**：上传截图、手绘草图、白板照片、设计效果图等。系统将图片通过配置的视觉模型 API（如 GPT-4o、Claude 3.5 Sonnet、Gemini 2.5 Pro 等）进行分析理解。典型场景：上传一张 UI 设计稿的截图，告诉 Agent“请根据这个设计生成对应的 React 组件代码”。Agent 会调用视觉模型理解设计稿中的布局、颜色、组件结构，然后生成对应的前端代码。

- **PDF 文档**：上传 PDF 文件。系统通过文档解析 API 提取文本、表格、图片等内容，交给 LLM 进行分析理解。典型场景：上传一份财报 PDF，Agent 提取关键财务数据，生成分析报告。

- **Office 文档**：支持 Word（.docx）、Excel（.xlsx）、PowerPoint（.pptx）等格式。同样通过云端文档解析 API 提取结构化内容。

- **代码包**：上传 .zip 或 .tar.gz 压缩包。Agent 解压到工作空间后进行代码分析、重构、或生成文档。

所有上传文件首先经过病毒扫描（如果系统配置了相关 API），然后暂存在 Agent 的沙箱工作区内。文件解析和理解的推理计算全部发生在云端，用户设备仅负责文件上传和结果展示。

**输出多模态——让 Agent 能说会画**：
Agent 的回复不局限于纯文本，可以生成丰富的内容形态：

- **图表与可视化**：Agent 可以在回答中生成 Mermaid 图表（流程图、序列图、甘特图、类图等）和 ECharts 图表（柱状图、折线图、饼图等）。这些图表在 Web 面板的消息区域中直接渲染，用户可以缩放、导出为 PNG/SVG。

- **HTML 页面**：Agent 可以生成完整的 HTML 页面（内联 CSS 和 JS），作为富媒体卡片在对话中直接展示。用户可以交互式地使用这个页面，或者下载完整文件。

- **代码块与语法高亮**：代码生成是编程类 Agent 的核心输出。Web 面板使用 highlight.js 对所有代码块进行语法高亮，支持 190+ 编程语言。代码块提供一键复制按钮，方便用户粘贴到编辑器中。

- **文件下载**：Agent 生成的文件（代码文件、文档、图片等）可一键下载到用户本地。

**语音交互——用说的就行**：
语音交互全部通过云端语音 API 实现，程序本身不携带任何语音识别或合成模型，保持二进制轻量：

- **语音输入（STT，语音转文字）**：Web 界面提供一个麦克风按钮。用户点击后，浏览器录制音频，将音频数据发送到后端，后端转发至配置的语音识别 API（如 OpenAI Whisper API、Azure Speech-to-Text、Deepgram 等）。返回的文本自动填入消息输入框，用户确认后发送给 Agent。整个音频处理在云端完成，用户端无需 GPU 或任何本地模型。

- **语音输出（TTS，文字转语音）**：Agent 的文字回复可以通过语音合成 API 转为自然语音。用户点击消息旁的“播放”按钮，后端将文本发送至配置的 TTS API（如 OpenAI TTS、Azure Text-to-Speech、ElevenLabs 等），返回的音频流在前端播放。用户可以在设置中选择语音（男声/女声）、语速、音色等。

语音功能的 API 凭证与 LLM 凭证统一在模型网关中管理。用户可以选择使用和 LLM 相同的服务商（如都用 OpenAI），也可以为语音单独配置服务商。


#### 2.13 高级性能与成本优化（云端 API 优化）

即使拥有充足的算力和 API 预算，良好的优化策略依然能显著提升用户体验（响应更快）和降低成本。所有优化策略均基于云端 API 的特性，不涉及本地模型。

**提示词缓存透明化（Prompt Caching）**：
这是成本优化最有效的单一手段。现代 LLM 提供商（OpenAI、Anthropic、DeepSeek 等）的 API 都支持某种形式的提示词缓存。其原理是：如果 API 请求的前缀部分与之前的某次请求完全相同，服务端可以复用之前的计算，对这部分 Token 仅收取极低的费用（通常是原价的 10%），同时生成首 Token 的速度也更快。

本系统在架构层面自动优化 Prompt 结构以最大化缓存命中率。核心做法是将相对固定的内容放在请求前缀：

- System Instruction（角色定义、行为准则、AGENTS.md 内容）
- 工具定义 Schema（工具名称、描述、参数定义）
- 近期加载的记忆文件内容
- 设定的项目规范和编码标准

而将动态变化的内容放在请求尾部：
- 用户最新输入的问题
- 前几步的工具调用结果
- 最新的对话轮次

系统自动按此顺序构建每次 LLM 请求的 Prompt。只要 Agent 的角色配置和工具集不变，每次请求的静态前缀就完全一致，LLM 提供商的缓存就能持续命中。用户无需任何额外操作，这个优化对用户完全透明。

**智能模型路由**：
不是每个请求都需要最强大（也是最贵）的模型。系统内置了一个轻量级的任务复杂度评估器，在每次发送 LLM 请求前，评估器会分析任务的预估难度，然后自动选择最合适的模型：

- **简单任务**（如“将这段 JSON 格式化”、“这个 Git 命令是什么”、“给这段代码加个注释”）→ 路由到快速且便宜的模型（如 GPT-4o-mini、Claude 3.5 Haiku）
- **中等任务**（如“写一个 CRUD 接口”、“分析这个 Bug 的可能原因”）→ 路由到中级模型（如 GPT-4o、Claude 3.5 Sonnet、DeepSeek-V3）
- **复杂任务**（如“为这个系统设计整体架构”、“审查所有代码找出安全漏洞”、“规划一个包含 10 个子步骤的部署方案”）→ 路由到强大模型（如 Claude Opus、o1-preview）

路由规则可以由用户自定义，也可以让系统自动学习。用户还可以为每个 Agent 设定模型预算上限，系统在预算范围内自动做最优分配。

**上下文压缩与摘要**：
在多轮对话或长链任务中，对话上下文会快速膨胀。如果不加控制，很快就会达到模型的上下文窗口上限，或者即使没到上限，也会因为大量历史内容导致每轮请求的 Token 消耗激增（成本成倍增长）。

系统会监控上下文长度，当接近阈值时（默认可配置），自动触发上下文压缩：
1. 调用一个小型/便宜的模型（如 GPT-4o-mini）对历史对话进行摘要
2. 摘要保留关键信息（用户目标、已完成的步骤、重要的中间结果、当前状态）
3. 将摘要替换冗长的原始历史，注入后续请求

压缩后的上下文保持了任务连续性所需的信息，同时将 Token 数从数万压缩到数百。用户可根据需要调整压缩的激进程度。

**工具调用优化**：
工具定义 Schema 也会消耗 Token。如果一个 Agent 拥有 30 个工具，每次 LLM 请求都要把这 30 个工具的完整定义（名称、描述、参数 Schema）塞进 Prompt，但实际可能只会用到 2-3 个。

系统采用按需动态加载策略：在发送 LLM 请求前，先评估当前任务最可能用到哪些工具，只将那部分工具的 Schema 注入 Prompt。如果 LLM 在推理中需要其他工具，可以在下一轮加载。

此外，对于 Agent 频繁执行的相同或高度相似的工具调用序列（如“拉取最新代码 → 安装依赖 → 运行测试 → 报告结果”），系统会自动识别这些模式，将其编译为确定性的脚本并缓存。后续遇到相同任务时，跳过 LLM 推理步骤，直接执行缓存脚本，既大幅降低了 Token 消耗，也消除了 LLM 可能产生的变异。

#### 3.2 Vue 3 前端技术栈

- **核心框架**：Vue 3.5+，使用 Composition API 和 `<script setup>` 语法，获得完整的 TypeScript 类型推断支持
- **构建工具**：Vite 7.x，开发环境实现即时代码热更新（HMR），生产构建基于 esbuild 和 Rollup，产物高度优化
- **UI 组件库**：Arco Design Vue（字节跳动开源的企业级 UI 组件库）。该组件库提供了丰富的组件，对 AI 对话场景有良好支持：消息列表（Chat）、卡片（Card）、标签页（Tabs）、表单（Form）、表格（Table）、抽屉（Drawer）、通知（Notification）等
- **状态管理**：Pinia 3.x，搭配 `pinia-plugin-persistedstate` 插件，将 Agent 配置、UI 主题偏好、面板布局等状态持久化到 localStorage
- **Markdown 富文本渲染**：
  - markdown-it 作为 Markdown 解析引擎
  - highlight.js 提供代码块语法高亮，支持 190+ 编程语言
  - Mermaid 渲染流程图、序列图、类图、甘特图等
  - KaTeX 渲染数学公式
  - 以上组合完整覆盖 Agent 可能输出的各类富文本内容
- **流式通信**：使用 SSE（Server-Sent Events）接收 Agent 的逐 Token 流式输出，前端以打字机效果实时渲染，用户可即时阅读无需等待完整回复
- **多模态内容展示**：图片预览组件（支持缩放、旋转）、音频播放器组件、Mermaid/ECharts 图表交互渲染组件
- **技能市场界面**：独立的单页应用模块，提供技能包的搜索（带全文检索）、分类树浏览、详情页展示（README 渲染）、一键安装与管理操作
- **静态资源嵌入**：前端项目执行 `vite build` 后生成的纯静态文件（HTML、CSS、JS、字体、图标），通过 Go 1.16+ 的 `//go:embed` 指令在编译时直接嵌入到二进制文件中。程序启动后，内置 Go HTTP 服务器直接将这些文件返回给浏览器，无 CDN 依赖

#### 3.3 后端关键技术

- **核心开发语言**：Go（Golang），版本 1.22+。选择 Go 的原因：编译为单一原生二进制文件、极快的启动速度、低内存占用、原生高并发支持、强大的交叉编译能力

- **Agent 框架**：基于 **Eino**（字节跳动开源的 Go 语言 AI Agent 框架）。Eino 为 Go 生态提供了最完善的 Agent 构建能力，本系统直接使用其内置组件：
  - `ChatModelAgent`：提供开箱即用的 ReAct（推理与行动）循环能力
  - `Supervisor`：多 Agent 层级调度，中心化的任务分发与协调
  - `Plan-Execute`：规划-执行-反思模式，支持计划的动态调整
  - `DeepAgent`：深度推理与复杂任务分解

- **Web 服务框架**：推荐使用 **Gin** 或标准库 `net/http`。Gin 提供了高性能的路由、中间件引擎和完善的第三方生态。路由设计为 RESTful 风格 API，同时提供 WebSocket 升级端点（用于实时双向通信）和 SSE 端点（用于流式响应推送）

- **通信协议支持**：
  - **HTTP RESTful API**：前端与后端的标准通信接口，涵盖 Agent CRUD、配置管理、对话管理、系统设置等
  - **SSE（Server-Sent Events）**：Agent 流式响应的推送通道，前端通过 EventSource API 接收逐 Token 更新
  - **WebSocket**：用于需要双向实时通信的场景（如 Agent 执行状态的实时推送）
  - **A2A（Agent-to-Agent）协议**：Agent 间协作的标准协议栈，包括能力发现、任务委托、状态同步
  - **MCP（Model Context Protocol）**：作为客户端接入外部工具服务器

- **数据存储**：
  - **SQLite**：作为主数据库，存储 Agent 实例配置、会话历史、API Key（AES 加密存储）、技能包元数据、审计日志等结构化数据。SQLite 无需独立数据库进程，一个文件即是一个数据库，完美契合单二进制交付理念
  - **向量扩展**（如 sqlite-vec）：在 SQLite 内实现向量存储和相似度检索，用于 Agent 长期记忆的语义搜索
  - **文件系统**：Markdown 记忆文件、Agent 工作空间、技能包缓存等非结构化数据直接以文件形式存储

- **多模态处理**：所有图片、文档、语音的识别与生成，均通过 LLM 网关统一调用已配置的云端 API 实现。网关层抽象了不同服务商的接口差异，对内提供统一的调用接口。支持的视觉模型：GPT-4o/4V、Claude 3.5 Sonnet、Gemini 2.5 Pro 等；语音模型：Whisper API、Azure Speech、ElevenLabs 等

- **安全与隔离**：
  - API Key 等敏感凭证使用 AES-256 对称加密存储于 SQLite
  - 支持 RBAC（基于角色的访问控制）：管理员与只读用户权限分离
  - 所有工具调用通过操作系统原生沙箱隔离执行（无 Docker 依赖）
  - 完整的审计日志记录，满足安全合规要求

- **跨平台构建**：
  - 通过 Go 的 `GOOS` 和 `GOARCH` 环境变量进行交叉编译
  - CI/CD 使用 GitHub Actions，利用 GoReleaser 自动构建多平台二进制、生成 Release Notes、校验和、并推送至 GitHub Releases

#### 3.4 关键设计原则

- **本地优先（Local-First）**：所有用户数据（配置、对话历史、记忆、工作空间）默认存储在用户本地设备的 XClaw 根目录下。Windows 为 `%APPDATA%\xclaw\`，macOS 为 `~/Library/Application Support/xclaw/`，Linux 为 `~/.xclaw/`。用户对自己的数据拥有绝对主权。系统可以在完全离线的局域网环境中启动和运行，仅需 LLM API 的网络连接

- **纯 API 驱动**：不捆绑、不内置任何本地模型权重文件。所有 AI 推理能力通过云端 API 实现。二进制文件保持小巧（目标 < 50MB），不随模型发展而膨胀

- **安全默认（Secure by Default）**：最小权限原则——高危操作（Shell 执行、文件删除、网络外连）默认要求用户显式确认或配置审批。工具安装默认失败关闭。沙箱默认开启

- **渐进式能力扩展**：核心引擎保持精简通用，不预设领域知识。所有专业技能通过技能市场、工具包、角色模板按需加载。用户按需安装，不用的功能不占资源

- **完全可观测**：Web 面板提供 Agent 实时状态、任务执行流水、Token 消耗图表、工具调用历史、错误日志等全维度可视化信息。用户始终了解 Agent 在做什么、花了多少钱、有没有问题


### 4. 部署与分发

- **单二进制交付**：用户从 GitHub Releases 页面下载对应操作系统和架构的单个二进制文件。macOS/Linux 上 `chmod +x agent && ./agent`，Windows 上双击运行。启动后浏览器访问 `http://localhost:8080`（可选开启 HTTPS）即可使用整个系统。无需任何额外的安装步骤或依赖安装

- **零外部运行时依赖**：不要求安装 Docker、Node.js、Python、CUDA、或其他任何运行时。沙箱使用操作系统内核内置的安全原语。LLM 推理通过网络 API 调用。数据库内嵌（SQLite）

- **数据目录**：所有 Agent 配置、工作空间、记忆文件、数据库文件、技能包缓存、审计日志均默认存储在 XClaw 根目录下，统一分层为：
  - `config/`：运行配置、凭据、证书、策略、锁文件
  - `data/`：SQLite 数据库、工作空间、上传文件、知识库索引、导出文件
  - `skills/`：内置技能、已安装技能、覆盖层
  - `cache/`：技能下载缓存、远程 catalog 缓存、解析缓存
  - `logs/`：daemon 日志与诊断日志
  - `run/`：PID、运行状态、临时会话状态
  - `tmp/`：临时下载、解压与中间文件
  目录结构清晰、自包含，用户可以整体打包、备份、迁移到新设备

- **系统服务模式**：支持 `--daemon` 标志启动为后台守护进程。程序自动注册为对应平台的系统服务（Linux systemd、macOS launchd、Windows Service）。开机自启，后台静默运行，用户随时通过浏览器访问

- **自动更新**：启动时异步检查 GitHub Releases 最新版本。检测到更新后 Web 界面顶部提示，用户一键点击下载新二进制文件，程序自动验证校验和、替换当前文件、优雅重启，当前进行的对话不受影响

- **配置灵活性**：支持通过环境变量、配置文件（XClaw 根目录下的 `config/config.json`）与命令行参数三种方式进行配置。优先级：命令行参数 > 环境变量 > 配置文件 > 默认值。旧版本的平铺式 `config.json/config.yaml` 需要在升级时自动迁移到新目录结构。


### 5. 典型使用场景

以下场景展示了本系统在实际工作中的使用方式和价值：

**场景一：全栈开发团队**
用户创建一个“技术主管”Agent 和三个工程师 Agent（前端 React、后端 Go、测试）。为每个 Agent 配置对应的角色描述和工具权限。用户在主管 Agent 的对话中输入：“开发一个用户管理系统，包括注册、登录、权限管理功能”。主管 Agent 自动分解需求 → 设计 API 接口规范 → 将前端任务分配给前端 Agent，后端任务分配给后端 Agent → 各 Agent 在各自沙箱工作空间中并行开发 → 测试 Agent 自动生成测试用例 → 主管汇总代码、运行自动化验证管道 → 将通过的完整项目交付给用户。整个过程用户只做了一次高层级描述。

**场景二：营销策划作战室**
用户组建“营销总监”、“文案策划”、“数据分析师”、“社交媒体运营”四个 Agent 组成营销团队。总监接收“为新产品上市策划一个为期两周的预热活动”的目标，自动生成策划方案并分派任务。文案 Agent 生成各平台宣传文案，运营 Agent 制定发布日程，分析 Agent 实时跟踪传播数据并指导优化策略。各 Agent 在独立空间中工作，通过团队内部的共享记忆同步进度和关键信息。

**场景三：个人全能助理**
用户只创建一个 Agent 实例。周一告诉它“帮我规划日本七日游，生成详细的行程单和预算表”（Agent 切换为旅行规划师角色）；周二上传一张财务表格的截图，说“分析这几项数据的变化趋势”（Agent 调用视觉模型理解截图，切换为数据分析师角色）；周三说“帮我写一封正式的辞职信，措辞要得体但坚定”（Agent 切换为高级文书专家）。同一个 Agent，同一套引擎，不同的角色指令，即时应答。

**场景四：自动化运维值班**
用户创建一个运维 Agent，配置 Shell 执行、SSH 连接、日志分析等工具，设置为定时任务模式。每天早上 8 点，Agent 自动通过 SSH 连接生产服务器，拉取监控指标和日志，分析是否存在异常。生成运维日报（Markdown 格式，包含关键指标趋势图），发布到团队的共享工作空间。如果发现高危指标（如磁盘使用率超过 90%），自动创建告警并将审批请求推送给管理员。

**场景五：企业级合规助手**
在金融或医疗等强合规行业，用户配置 Agent 的审批规则：所有涉及生产数据库的操作需要主管审批，所有代码入库前必须通过安全扫描。Agent 在开发流程中自动执行 Lint、类型检查、单元测试、安全漏洞扫描。如果代码不合规，Agent 自动给出修改建议并要求开发 Agent 重写。涉及数据库变更的操作，Agent 生成变更脚本后暂停，向主管推送审批请求，主管在手机或电脑上确认后，Agent 才执行变更。

**场景六：跨组织虚拟协作**
用户 A 在她的设备上部署了一个“数据科学 Agent”，擅长数据分析。用户 B 在他的设备上部署了一个“可视化 Agent”，擅长生成图表和报告。通过 A2A 协议，两个 Agent 相互发现对方的能力。用户 A 的数据科学 Agent 完成数据分析后，自动将结果委托给用户 B 的可视化 Agent 生成专业图表，并将图表回传给用户 A。跨设备、跨部署实例的协作自动完成。

**场景七：智能个性化教育**
一个学生创建了一个“学习导师”Agent，上传了课程大纲和教材 PDF。Agent 索引这些材料后，根据学生的学习进度自动生成个性化练习题。学生作答后，Agent 批改并详细指出错误原因。学生可以用语音提问（通过云端 Whisper API 识别），Agent 用语音回答（通过云端 TTS 合成）。随着时间推移，Agent 记住学生的学习曲线和薄弱知识点，动态调整教学计划。
