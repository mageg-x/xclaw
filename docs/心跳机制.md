## 补充文档：心跳巡检 & 定时调度系统（完整版）

本文档是对《超级万能AI Agent 产品设计文档（v2.3）》的重要补充，聚焦 **Heartbeat（心跳巡检）** 与 **Cron（定时任务）** 两大核心自动化能力。该能力将 Agent 从“被动应答式工具”升级为“主动服务型伙伴”，是 Agent 迈向全自主运作的**分水岭级功能**。

> **重要声明**：心跳与 Cron 机制在 OpenClaw、Harness 等头部 AI Agent 产品中已被验证为关键成功因素。为避免后续出现能力遗漏，本系统明确将此项列为 **P0 优先级**，不允许任何形式的删减或弱化。


### 1. 背景与问题陈述

**现有架构的局限性**：
当前系统设计中，Agent 严格遵循“用户输入 → Agent 响应”的被动模式。一旦用户离线或未主动发起对话，Agent 便完全静止，无法执行任何自主性工作。这导致以下能力缺失：

- 无法主动监控外部事件（如邮件、日历、服务器状态）
- 无法在指定时间点向用户推送提醒或报告
- 无法在空闲时进行自我维护（如记忆整理、知识库更新）

**竞品对标与功能对齐**：
通过竞品全特性矩阵审查发现，OpenClaw 与 Harness 均通过“心跳 + 定时任务”的组合机制解决了上述问题。补齐此项后，本系统将达到行业最前沿水平。

| 核心特性 | OpenClaw | Harness | **本系统（当前）** |
| :--- | :---: | :---: | :---: |
| 心跳检查 (Heartbeat) | ✅ | 部分 | ✅ 已补齐（P0） |
| 定时任务 (Cron) | ✅ | ✅ | ✅ 已补齐（P0） |
| 社交通信集成 | ✅ | ❌ | ✅ 已补齐 |
| 多 Agent 协作 | ✅ | ✅ | ✅ |
| 文件化角色配置 | ✅ | ❌ | ✅ |


### 2. 设计目标

1.  **主动巡检**：Agent 可按照预设间隔自动执行检查清单，发现异常时主动通知用户。
2.  **精确定时**：支持自然语言描述的一次性提醒、固定间隔任务、标准 Cron 表达式。
3.  **成本可控**：心跳设计充分利用 Prompt Cache，避免无意义 Token 消耗；Cron 任务按需执行，不浪费资源。
4.  **通道通达**：所有主动推送的消息，均可通过已集成的社交通信工具（Telegram、微信、飞书等）触达用户。
5.  **简易管理**：定时任务可通过自然语言由 Agent 自行创建，也可通过 Web 面板进行增删改查。


### 3. Heartbeat（心跳巡检）详细设计

#### 3.1 核心概念

心跳是 Agent 的“自主呼吸”——一个轻量级的周期性检查循环。它不承担复杂的推理任务，只负责扫描用户预设的检查清单，并在发现问题时触发通知。

#### 3.2 关键参数与默认值（对齐 OpenClaw）

| 参数 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `enabled` | `true` | 是否启用心跳 |
| `interval` | `30m` | 心跳间隔。详细说明见 3.3 节 |
| `active_hours.start` | `08:00` | 心跳活跃时段开始（避免夜间打扰） |
| `active_hours.end` | `22:00` | 心跳活跃时段结束 |
| `target` | `last` | 通知推送通道（`last` / 具体通道名） |
| `max_rounds` | `5` | 单次心跳内最多执行工具调用的轮数（防止死循环） |
| `session_idle_timeout` | `8h` | 用户连续不互动超过此时长，触发问候消息 |

#### 3.3 默认 30 分钟间隔的深度理由（全面对标 OpenClaw）

30 分钟作为默认值，是经过头部产品工程验证的“黄金平衡点”，理由如下：

1.  **成本优化与 Cache 保持**：OpenClaw 深度依赖 Prompt Cache（尤其是 Anthropic 的缓存机制）来降低 Token 成本。心跳间隔若过长（>1 小时），缓存极易过期失效，导致下一次心跳变成昂贵的“冷启动”。30 分钟间隔可将心跳请求的前缀稳定性保持在极高水位，缓存命中率可达 90% 以上，使每次心跳的 API 成本降至极低。
2.  **响应实时性**：对于“30 分钟内需要处理的会议提醒”“紧急邮件检查”等场景，30 分钟是可以接受的完成时延。若延长至 1 小时，将错过及时的通知窗口。
3.  **队列漂移容错**：心跳并非绝对准点秒级触发，它会因主会话的实际负载产生几秒到几分钟的自然漂移。这种不精确性天然避免了大量 Agent 在同秒并发请求模型，形成了一种隐形的负载均衡。
4.  **凭证自适应**：当系统检测到用户使用 Anthropic OAuth/Token 凭证（或有严格并发限制的 API Key）时，心跳间隔会自动放宽至 **1 小时**，以符合服务商的配额策略。此项兼容设计与 OpenClaw 完全一致。

**可扩展性**：未来可引入“自适应心跳 (Adaptive Heartbeat)”，根据用户当前在线状态、任务紧急程度等信号，在 5 分钟到 2 小时之间动态调整频率，进一步优化资源消耗。

#### 3.4 工作流程

1.  **触发器启动**：调度器根据 `interval` 周期性触发心跳事件。
2.  **检查清单加载**：心跳进程读取 Agent 工作空间中的 `HEARTBEAT.md` 文件。
3.  **逐项执行**：Agent 将检查清单作为系统指令，进行一次“思考-行动-观察”循环。它会调用必要的工具（如邮箱 API、日历 API、系统监控 API）逐项核实。
4.  **结果判定**：
    - **无事发生**：所有检查项正常，Agent 返回 `HEARTBEAT_OK` 静默信号。**此时不推送任何消息给用户，也不产生额外 Token 消耗（除一次几乎全命中的 Cache 读取外）**。
    - **发现问题**：Agent 生成一条精炼的告警或提醒消息，通过配置的 `target` 通道主动推送给用户。
5.  **日志记录**：无论有无发现，心跳执行记录（时间、检查项、结果）均写入后台任务日志，便于审计。

#### 3.5 HEARTBEAT.md 模板（用户可编辑）

```markdown
# 每日自动化检查清单

- **邮件检查**：检索收件箱中最近 2 小时内的未读邮件，若标题含“紧急”、“重要”、“催办”等词，立刻通知我并附上摘要。
- **日历提醒**：检查我接下来 2 小时内的日历日程，提前 15 分钟提醒我。
- **待办扫描**：检索我的待办事项列表，若有已逾期或今天内截止的任务，汇总后告知。
- **服务监控**：若我已配置上线服务的健康检查端点，请探测其可用性。若返回 5xx 或超时，立即告警。
- **用户问候**：若我连续 8 小时未与你互动，发送一条友好的问候，并询问是否需要帮助。
- **日报检查**：每个工作日下午 17:30，检查我今日是否已完成工作日志或日报填写。若未完成，进行一次提醒。
```

#### 3.6 心跳执行器实现细节（新增）

心跳并非普通的 Cron 任务，而是一个**轻量化的专用执行器**，核心设计如下：

```go
// heartbeat/runner.go
package heartbeat

type Runner struct {
    engine   *engine.Service      // 复用现有 Agent 引擎
    ws       *workspace.Manager   // 读取 HEARTBEAT.md
    gateway  *gateway.Gateway     // 推送告警通知
    monitor  *llm.TokenMonitor    // 记录 Token 消耗
    audit    *audit.Logger        // 审计日志
    store    *db.Store            // 数据存储
}

func (r *Runner) Run(ctx context.Context, agentID string) error {
    // 1. 读取 HEARTBEAT.md，若不存在则跳过本次心跳
    hbFile := filepath.Join(r.ws.GetAgentPath(agentID), "HEARTBEAT.md")
    checklist, err := os.ReadFile(hbFile)
    if err != nil {
        return nil // 静默跳过，不报错
    }

    // 2. 构建极简 Prompt，确保前缀稳定以命中 Prompt Cache
    prompt := fmt.Sprintf(`<system>
You are the heartbeat monitor for Agent %s.
Checklist:
%s
</system>
Current time: %s
Execute the checklist. If everything is OK, respond with exactly "HEARTBEAT_OK" and nothing else.
If you find any issue, respond with a concise alert message (max 200 chars).`,
        agentID, string(checklist), time.Now().Format(time.RFC3339))

    // 3. 创建临时会话并调用 LLM，限制 max_rounds=5
    // 复用现有 engine.SendMessage 方法，而非不存在的 RunHeartbeat
    session, err := r.engine.CreateSession(ctx, agentID, "Heartbeat Check", false)
    if err != nil {
        return fmt.Errorf("create heartbeat session: %w", err)
    }
    defer r.store.DeleteSession(ctx, session.ID) // 执行完后销毁临时会话

    result, err := r.engine.SendMessage(ctx, session.ID, engine.SendMessageInput{
        Content:     prompt,
        AutoApprove: true, // 心跳任务自动审批
    })
    if err != nil {
        return fmt.Errorf("heartbeat llm call: %w", err)
    }

    // 4. 判定结果
    reply := result.Reply
    if strings.TrimSpace(reply) == "HEARTBEAT_OK" {
        // 静默成功，仅记录日志
        r.audit.Log(ctx, agentID, "", "heartbeat", "ok", "")
        return nil
    }

    // 5. 发现问题，推送通知（使用新版 OutboundEvent）
    target := r.resolveTarget(agentID)
    evt := gateway.OutboundEvent{
        Platform:       parsePlatform(target),
        ChatID:         parseChatID(target),
        ThreadID:       parseThreadID(target),
        TextMarkdown:   reply,
        Priority:       "high",
        IdempotencyKey: fmt.Sprintf("hb:%s:%d", agentID, time.Now().Unix()/60), // 按分钟粒度幂等
        TTLSeconds:     3600,
    }
    return r.gateway.Send(ctx, evt)
}

// resolveTarget 解析目标地址，支持 last / platform:chat_id / binding:id 格式
func (r *Runner) resolveTarget(agentID string) string {
    // 从 Agent 配置或全局配置获取默认目标
    // 实际实现需读取配置文件或数据库
    return "last"
}

func parsePlatform(target string) string {
    parts := strings.Split(target, ":")
    if len(parts) >= 2 && parts[0] != "last" && parts[0] != "binding" {
        return parts[0]
    }
    return ""
}

func parseChatID(target string) string {
    parts := strings.Split(target, ":")
    if len(parts) >= 2 && parts[0] != "last" && parts[0] != "binding" {
        return parts[1]
    }
    return ""
}

func parseThreadID(target string) string {
    parts := strings.Split(target, ":")
    if len(parts) >= 3 && parts[0] != "last" && parts[0] != "binding" {
        return parts[2]
    }
    return ""
}
```

**关键实现要点**：
- **Prompt 前缀稳定**：`<system>` 标签内的 Agent ID 和 Checklist 内容保持不变，只有 `Current time` 动态变化，最大化 Cache 命中率。
- **静默成功**：`HEARTBEAT_OK` 时不产生任何外部通知，避免打扰用户。
- **轻量推理**：心跳不执行复杂的多步 ReAct，仅允许最多 5 轮工具调用，防止资源耗尽。
- **缺失文件容错**：若用户未创建 `HEARTBEAT.md`，心跳自动跳过，不报错、不通知。

#### 3.7 安全与省电考量

- **活跃时段限制**：默认仅在工作时间（08:00-22:00）启用心跳，防止深夜推送打扰。
- **防抖机制**：同一 Agent 同一时间只允许一个心跳实例运行，防止并发冲突。
- **优雅关闭**：Agent 进程退出时，等待进行中的心跳完成后才终止。


### 4. Cron（定时任务）详细设计

#### 4.1 核心概念

Cron 是 Agent 的“精准闹钟”，用于执行准确性要求高、时间点明确的任务。它与心跳互补：**心跳负责模糊周期性巡检，Cron 负责精确定时执行**。

#### 4.2 任务类型

| 类型 | 解析示例 | 执行行为 |
| :--- | :--- | :--- |
| `at` | “明天 9 点”、“20 分钟后” | 在指定时间执行一次，完成后自动标记为 `completed`，保留记录。 |
| `every` | “每 2 小时”、“每 30 分钟” | 每过指定时间间隔重复执行。 |
| `cron` | “0 9 * * *” | 标准 5/6 字段 Cron 表达式，精确控制分钟、小时、日期、月份、星期。 |

#### 4.3 任务类型解析实现（新增）

现有代码中的 `scheduler/cronexpr.go` 仅支持标准 5 字段 Cron 表达式。为支持 `at` 与 `every` 类型，调度器需扩展如下：

```go
// scheduler/cronexpr.go（扩展）
package scheduler

import (
    "time"
    "github.com/robfig/cron/v3"
)

type cronExpr struct {
    spec string
    cron cron.Schedule
}

func parseCron(spec string) (*cronExpr, error) {
    // 支持标准 5 字段和 6 字段（秒级）表达式
    parser := cron.NewParser(cron.Minute | cron.Hour | cron.Dom | cron.Month | cron.Dow)
    schedule, err := parser.Parse(spec)
    if err != nil {
        return nil, err
    }
    return &cronExpr{spec: spec, cron: schedule}, nil
}

func (e *cronExpr) match(t time.Time) bool {
    next := e.cron.Next(t.Add(-time.Minute))
    return next.Sub(t) < time.Minute
}

// Next 返回下一个触发时间（新增方法）
func (e *cronExpr) Next(t time.Time) time.Time {
    return e.cron.Next(t)
}
```

```go
// scheduler/job.go
package scheduler

type JobType string

const (
    JobTypeAt    JobType = "at"
    JobTypeEvery JobType = "every"
    JobTypeCron  JobType = "cron"
)

// resolveNextRun 计算下一次执行时间，支持时区处理
func (s *Scheduler) resolveNextRun(job models.CronJob) (time.Time, error) {
    // 获取任务时区，默认使用系统时区
    loc := time.Local
    if job.Timezone != "" {
        var err error
        loc, err = time.LoadLocation(job.Timezone)
        if err != nil {
            return time.Time{}, fmt.Errorf("invalid timezone %s: %w", job.Timezone, err)
        }
    }

    switch JobType(job.JobType) {
    case JobTypeAt:
        // schedule 为 ISO8601 格式，如 "2026-04-25T09:00:00"
        // 先尝试 RFC3339 格式（带时区）
        t, err := time.Parse(time.RFC3339, job.Schedule)
        if err != nil {
            // 再尝试本地时间格式（需要结合时区）
            t, err = time.ParseInLocation("2006-01-02T15:04:05", job.Schedule, loc)
            if err != nil {
                return time.Time{}, fmt.Errorf("invalid at schedule: %w", err)
            }
        }
        return t, nil

    case JobTypeEvery:
        // schedule 为 duration 格式，如 "30m", "2h"
        d, err := time.ParseDuration(job.Schedule)
        if err != nil {
            return time.Time{}, fmt.Errorf("invalid every duration: %w", err)
        }
        // 从 last_run_at 计算下一次（从开始时间计算，避免漂移）
        if job.LastRunAt != nil {
            return job.LastRunAt.Add(d), nil
        }
        return time.Now().In(loc).Add(d), nil

    case JobTypeCron:
        // 复用现有 parseCron + 扩展 Next() 方法
        expr, err := parseCron(job.Schedule)
        if err != nil {
            return time.Time{}, fmt.Errorf("invalid cron expression: %w", err)
        }
        // 在指定时区计算下一次触发时间
        return expr.Next(time.Now().In(loc)), nil
    }

    return time.Time{}, fmt.Errorf("unknown job type: %s", job.JobType)
}
```

**时区处理说明**：

1. **存储格式**：所有时间在数据库中以 UTC 格式存储。
2. **解析规则**：
   - `at` 类型：若 schedule 不含时区信息，使用任务配置的 `timezone` 字段解析。
   - `every` 类型：间隔计算不受时区影响，但首次执行时间基于当前时区。
   - `cron` 类型：Cron 表达式在指定时区下计算触发时间。
3. **默认时区**：若任务未指定 `timezone`，继承系统时区（`time.Local`）。
4. **用户时区获取**：可通过 Agent 配置、用户 Profile 或浏览器时区推断。

#### 4.5 执行模式

- **`main`（主会话）**：任务作为一条系统消息注入 Agent 的主会话泳道。Agent 将在其熟悉的上下文中处理，适合需要结合用户历史记忆的任务。
- **`isolated`（隔离会话）**：创建一个临时独立会话 `cron:<jobId>`，任务在该纯净上下文中执行，完成后立即销毁，不污染主对话历史。适合简单的查询和通知类任务。
- **`custom`（自定义会话）**：绑定到用户指定的持久化会话，多次执行之间可以保留状态，适合需要跨任务积累数据的场景。

**执行模式分发逻辑（新增）**：

```go
// scheduler/dispatcher.go
package scheduler

import (
    "context"
    "fmt"
    "xclaw/cli/db"
    "xclaw/cli/engine"
    "xclaw/cli/models"
)

type Dispatcher struct {
    store  *db.Store
    engine *engine.Service
}

func (d *Dispatcher) dispatch(ctx context.Context, job models.CronJob) error {
    switch job.ExecutionMode {
    case "main", "":
        // main 模式：获取或创建主会话，注入系统消息
        sessions, err := d.store.ListSessions(ctx, job.AgentID)
        if err != nil {
            return fmt.Errorf("list sessions: %w", err)
        }
        var mainSession *models.Session
        for _, s := range sessions {
            if s.IsMain {
                mainSession = &s
                break
            }
        }
        if mainSession == nil {
            // 创建主会话
            mainSession, err = d.engine.CreateSession(ctx, job.AgentID, "Main", true)
            if err != nil {
                return fmt.Errorf("create main session: %w", err)
            }
        }
        // 复用现有 SendMessage 方法注入任务内容
        _, err = d.engine.SendMessage(ctx, mainSession.ID, engine.SendMessageInput{
            Content:     job.Payload,
            AutoApprove: true, // Cron 任务自动审批
        })
        return err

    case "isolated":
        // isolated 模式：创建临时会话，执行后销毁
        session, err := d.engine.CreateSession(ctx, job.AgentID, "Cron: "+job.Name, false)
        if err != nil {
            return fmt.Errorf("create isolated session: %w", err)
        }
        defer d.store.DeleteSession(ctx, session.ID) // 执行完后销毁
        _, err = d.engine.SendMessage(ctx, session.ID, engine.SendMessageInput{
            Content:     job.Payload,
            AutoApprove: true,
        })
        return err

    case "custom":
        // custom 模式：使用指定的持久化会话
        if job.SessionID == "" {
            return fmt.Errorf("custom mode requires session_id")
        }
        _, err := d.engine.SendMessage(ctx, job.SessionID, engine.SendMessageInput{
            Content:     job.Payload,
            AutoApprove: true,
        })
        return err

    default:
        return fmt.Errorf("unknown execution mode: %s", job.ExecutionMode)
    }
}
```

**关键实现要点**：
- **复用现有方法**：使用 `engine.CreateSession` 和 `engine.SendMessage`，而非不存在的 `InjectSystemMessage` / `RunSession` / `GetMainSession`。
- **主会话查找**：通过 `store.ListSessions` 遍历查找 `IsMain=true` 的会话。
- **异步安全**：`isolated` 模式的 `defer DeleteSession` 在 `SendMessage` 返回后执行，此时消息已处理完毕。

#### 4.6 自然语言解析 → 定时任务

Agent 通过内置工具 `add_cron_job` 来完成从自然语言到任务配置的转换。用户在对话中说：

> “明天早上 9 点提醒我看上周的销售报告。”

Agent 会执行类似以下推理和行动：
1.  **思考**：需要创建一个定时任务，类型为 `at`，目标时间为“明天 9:00”，内容为“提醒用户查看上周的销售报告”。
2.  **行动**：调用工具 `add_cron_job`，参数为 `{schedule: "2026-04-25T09:00:00", type: "at", payload: "提醒用户查看上周的销售报告", execution_mode: "main"}`。
3.  **确认**：工具执行成功后，Agent 回复：“已设置提醒：明天 9:00 准时提醒您查看上周销售报告。”

同理，用户说：“每周五下午 5 点帮我生成项目进度总结，并发送到飞书。”

Agent 会创建一个 `cron` 任务：`0 17 * * 5`，内容为“生成项目进度总结并推送到飞书”。

#### 4.7 持久化与故障恢复

所有定时任务持久化存储在 `~/.agent/cron/jobs.db`（SQLite 表 `cron_jobs`）中。程序意外退出或重启，调度器会立即从数据库恢复所有任务，并重新计算下一次触发时间，确保**不丢失、不错过**。

### 6. 与社交通信网关的闭环（优化版）

心跳和 Cron 统一通过通信网关发送，不再区分“业务通知链路”和“社交通道链路”。

**统一寻址格式**：
- `last`
- `platform:chat_id`
- `platform:chat_id:thread_id`
- `binding:<binding_id>`

**路由优先级**：
1. 任务级 `target_channel`（最优先）
2. Agent 级默认目标
3. 全局默认目标
4. `last`（最近活跃绑定）

**可靠性要求（必须）**：
1. 同一通知必须携带 `idempotency_key`，避免跨平台重发。
2. 推送失败按指数退避重试，并写入审计日志。
3. 平台不可用时自动降级到备用通道（如 `feishu -> telegram -> email`）。
4. 夜间静默窗口生效时，低优先级消息延期发送，高优先级告警可越过静默策略。

**统一发送示例**：

```go
// scheduler/notifier.go
package scheduler

import (
    "context"
    "fmt"
    "time"

    "xclaw/cli/gateway"
)

type Notifier struct {
    gateway *gateway.Gateway
    store   *db.Store
}

// notify 统一通知发送，支持幂等和降级
func (n *Notifier) notify(ctx context.Context, agentID string, jobID string, content string, target string, priority string) error {
    resolved := n.resolveTarget(agentID, target) // last / platform:chat / binding:id
    evt := gateway.OutboundEvent{
        Platform:       parsePlatform(resolved),
        ChatID:         parseChatID(resolved),
        ThreadID:       parseThreadID(resolved),
        TextMarkdown:   content,
        Priority:       priority, // low/normal/high/critical
        IdempotencyKey: fmt.Sprintf("cron:%s:%s:%d", agentID, jobID, time.Now().Unix()/60), // 按分钟粒度幂等
        TTLSeconds:     3600,
    }
    return n.gateway.Send(ctx, evt)
}

// resolveTarget 解析目标地址
func (n *Notifier) resolveTarget(agentID string, target string) string {
    if target == "" || target == "last" {
        // 从 Agent 配置获取最近活跃通道
        binding, err := n.store.GetLastActiveBinding(context.Background(), agentID)
        if err == nil && binding != nil {
            return fmt.Sprintf("%s:%s", binding.Platform, binding.ChatID)
        }
    }
    return target
}
```

**IdempotencyKey 设计说明**：

1. **格式**：`{类型}:{agent_id}:{job_id}:{时间戳}`
   - 类型前缀区分来源：`cron`（定时任务）、`hb`（心跳）、`approval`（审批）
   - `job_id` 标识具体任务，重试时保持不变
   - 时间戳按分钟取整（`Unix()/60`），同一分钟内的重试共享同一幂等键

2. **幂等保证**：
   - 网关收到相同 `IdempotencyKey` 的请求时，直接返回成功，不重复发送
   - 幂等键有效期 24 小时（通过 Redis 或内存缓存实现）

3. **降级策略**：
   - 主平台发送失败时，自动尝试备用平台
   - 所有平台失败时，写入死信队列（DLQ）等待人工处理


### 7. 典型场景闭环演示

**场景**：“明天 9 点提醒我看上周的销售报告”

### 9. Prompt Cache 优化策略（新增）

心跳是高频调用场景，Prompt Cache 命中率直接决定成本。系统采用以下策略确保高命中率：

1. **前缀稳定原则**：心跳请求的 System Prompt 必须保持字节级一致，包括：
   - Agent 角色定义
   - HEARTBEAT.md 检查清单内容
   - 可用工具列表及其 JSON Schema
   只有 `Current time` 等动态信息放在 Prompt 尾部。

2. **模型适配**：
   - **Anthropic Claude**：利用其官方 Prompt Cache API，在请求头中标记 `cache_control`。
   - **OpenAI / 其他模型**：通过保持前缀一致，依赖模型内部的 KV-Cache 复用机制。

3. **命中率监控**：
   ```go
   type CacheMetrics struct {
       AgentID        string    `json:"agent_id"`
       TotalCalls     int       `json:"total_calls"`
       CacheHits      int       `json:"cache_hits"`
       HitRate        float64   `json:"hit_rate"`
       AvgTokensSaved int       `json:"avg_tokens_saved"`
   }
   ```
   每次心跳执行后记录缓存命中情况，低于 80% 时触发告警，提示检查 Prompt 稳定性。

4. **降级策略**：当检测到缓存命中率持续低于阈值时，自动放宽心跳间隔至 1 小时，减少冷启动频率。


### 10. 与现有调度器的整合说明（新增）

现有代码中的 `scheduler/scheduler.go` 已实现基于 10 秒 tick 的 Cron 调度器，支持标准 5 字段表达式。心跳与 Cron 的整合方式如下：

1. **Cron 调度器复用**：标准 `cron` 类型任务继续使用现有调度器，未来可平滑迁移至 `robfig/cron/v3`。
2. **心跳独立模块**：心跳在 `heartbeat/` 包中独立实现，拥有独立的 30 分钟 ticker，不混用 Cron 调度器，避免互相干扰。
3. **统一数据库**：心跳日志与 Cron 任务共用 SQLite 存储，便于统一查询与审计。
4. **共享 LaneQueue**：心跳与 Cron 任务的执行均通过 `queue.LaneQueue` 入队，利用现有并发调度能力，避免资源争抢。

```
┌─────────────────────────────────────────────┐
│  Heartbeat Runner (30m ticker)              │
│  - 读取 HEARTBEAT.md                        │
│  - 轻量 LLM 调用 (max_rounds=5)             │
│  - 结果判定与通知推送                        │
├─────────────────────────────────────────────┤
│  Cron Scheduler (10s tick / robfig/cron)    │
│  - at / every / cron 类型任务调度            │
│  - 执行模式分发 (main/isolated/custom)       │
├─────────────────────────────────────────────┤
│  queue.LaneQueue (复用现有)                  │
│  - 统一并发控制与资源隔离                    │
└─────────────────────────────────────────────┘
```
